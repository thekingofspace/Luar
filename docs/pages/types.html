Type system

LUAR is dynamically typed. Annotations are parsed for the editor and then scrubbed before execution.

local name: string = "luar"
function area(w: number, h: number): number return w * h end
local x: Folder = nil :: any      -- a cast; the annotation wins

Type functions

Type functions are written name<T> and transform a type. The editor's type solver evaluates them, the grammar colors them, and completion offers each one as a name<...> snippet in any type position. This is the complete built-in set.

How the solver resolves them: most type functions belong to the unwrap family and resolve to their inner type argument, so a value annotated with one still completes and hovers like that inner type. The key family (keyof, nameof, indexof) resolves to string.

Indexed access

One type function takes two arguments instead of one. valuefrom reads the type of a value out of a type by key, which is useful for building your own typed lookups and import helpers.

Type functionMeaningResolves to
valuefrom<T, key>The type of the value stored at key inside the type T (a table or class type). For example valuefrom<Config, "port"> is the type of the port field. The editor resolves this to the container type T as a best effort.T

Because the unwrap family resolves to the inner type, a value annotated with one completes like that type:

local f: readonly<Folder> = openFolder()
f.        -- completes Folder's members; readonly<Folder> resolved to Folder

Type functions nest, resolving from the inside out:

type Inner = readonly<classof<Folder>>   -- resolves to Folder
type Key   = keyof<Config>               -- resolves to string
Type functions, like all annotations, are scrubbed before the program runs. They exist only to help the editor and the type solver.
