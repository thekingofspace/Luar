Precompilation

Precompilation is optional. LUAR runs .luar source directly — you never have to precompile anything to use any feature. It exists for when you want to load or ship a program without its source text, or skip parsing at startup. If you only ever run source, you can ignore this page.

What a .luarb is

A .luarb file is a precompiled program. The source has been tokenized, parsed, optimized, and had its type annotations scrubbed, and the resulting syntax tree is packed into a compact binary. It is not machine code and not a separate runtime: it is the very same program the parser would hand to the interpreter, frozen to bytes. Loading one skips the lexer and parser, and from there everything runs on the ordinary interpreter — classes, enums, mixins, operators, and the standard library all behave exactly as they do from source. Line numbers are kept in the binary, so runtime errors still point at the right line of the original source.

Producing a .luarb

Precompiled files are produced through the Rust API — the runner has no compile command. luar::precompile_source parses, optimizes, and packs source into bytes; you decide where the bytes live — a file on disk, an asset embedded in your binary, or a payload over the network.

let src   = std::fs::read_to_string("game.luar")?;
let bytes = luar::precompile_source(&src)?;     // Vec<u8>, the packed program
std::fs::write("game.luarb", &bytes)?;          // optional: save it somewhere

This is a thin wrapper over precompile::pack (syntax tree to bytes); its inverse precompile::unpack (bytes back to a syntax tree) is also public, so you can work directly with a parsed program if you already hold one.

Running a .luarb

A .luarb runs anywhere source runs.

  • The runner. luar-runtime game.luarb runs it exactly like running the source. The extension may be omitted — luar-runtime game tries game.luar then game.luarb — and pointing the runner at a folder looks for main.luar, main.luarb, init.luar, then init.luarb inside it.
  • require. Module resolution falls back from .luar to .luarb automatically (including init.luarb for folder modules), so a precompiled module is a drop-in replacement for its source — see Modules.
  • From Rust. Load the bytes with the host API, below.

From Rust

Several entry points load packed bytes, depending on what you want back:

let bytes = std::fs::read("game.luarb")?;

let interp = luar::run_precompiled(&bytes)?;            // run in a fresh interpreter
let (interp, returned) = luar::run_precompiled_returns(&bytes)?;  // + top-level return values
let module = luar::load_precompiled_module(&bytes)?;    // just the first returned value

let mut host = luar::Interpreter::new();
let results = host.run_precompiled(&bytes)?;            // run on an existing interpreter

load_precompiled_module is handy for module-style assets: it runs the program and hands you its first returned value (or nil), ready to install with set_global.

The optimizer is baked in

The same AST optimizer that the runtime applies to source (constant loop unrolling and friends) runs when a program is packed, so a .luarb already contains the optimized program — and it runs again when the bytes are loaded, so a file packed by an older LUAR still benefits from optimizations added since. There is no flag to turn this on; it is part of every load path.

Format compatibility

Every .luarb starts with a magic header that includes a format version. A loader only accepts files written with the exact format it understands; anything else — a corrupt file, or a file packed by a LUAR with a different format version — is rejected with not a LUAR precompiled file (bad header). After upgrading LUAR across a format bump, re-precompile your .luarb files from source. The format is otherwise portable: the same bytes load on any platform.

When to reach for it

  • Ship without source. Hand out .luarb bytes when you would rather not distribute the original .luar text.
  • Faster startup. Large programs skip lexing and parsing on load, which adds up when a host boots many scripts at once.
  • Embedding. Bake the bytes straight into your host binary and run them with run_precompiled, no files required.

Because annotations are scrubbed when the program is packed, a .luarb carries no type information; checking happens earlier, on the source, with Ferrite and the editor.
