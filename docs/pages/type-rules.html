Type rules (for tooling)

This page is for people building editor tooling, language servers, linters, or type solvers for LUAR. It describes exactly what type syntax the language accepts, which symbols are allowed and in what order they bind, and what happens to all of it at run time.

LUAR is dynamically typed. Every piece of type syntax below is parsed and then scrubbed before execution: the interpreter throws the annotations away and they have zero runtime effect. They exist only so tooling can read them. A program with all annotations deleted runs identically.

Where types may appear

The parser accepts a type in each of these positions and then discards it. A tool should expect a full type expression (see the grammar below) in every slot.

PositionExample
Variable bindinglocal x: number = 1 (also const and bare names)
Assignment targeta: number, b: string = 1, "x" (each target may carry a type)
Function parameterfunction f(a: number, b: string)
Vararg parameterfunction f(...: number)
Function returnfunction f(): boolean
Loop variablefor i: number = 1, 10 do (and generic-for names)
Class fieldpublic name: string = ""
Buff declarationbuff 16 buf: string = ""
Type aliastype Id = number (also export type)
Generic parameter listtype Box<T>, function f<T>(x), class C<T>
Castvalue :: SomeType

The type alias and the cast operator :: are themselves scrubbed: a type declaration produces no runtime value, and a cast evaluates only its left-hand expression and ignores the type. The cast is a postfix on an expression, so it chains like any other postfix: (t.field :: Handler)(x). An alias may be prefixed with export (export type Name = ... — note that pub type is not accepted); the prefix parses but the alias statement records no visibility, so exported and plain aliases are currently identical to tooling.

Binding order

Type operators bind from loosest to tightest in this order. A correct parser must layer them the same way so it agrees with what the language scrubs.

LevelOperatorFormAssociativity
1 (loosest)Function arrowParams -> Retright
2UnionA | B | Cflat (n-ary)
3IntersectionA & Bflat (n-ary)
4OptionalT? (postfix, repeatable)postfix
5 (tightest)Atomsnames + <...>, literals, {...}, (...)—

So the same way the parser groups them:

A | B & C?        -- A | (B & (C?))
A | B -> C        -- (A | B) -> C   (the arrow is loosest)
A -> B -> C     -- A -> (B -> C)  (right associative)
T??              -- optional of optional; ? may repeat

Grammar

The complete shape the parser accepts, loosest rule first:

type          = union ( "->" type )?
union         = intersection ( "|" intersection )*
intersection  = postfix ( "&" postfix )*
postfix       = primary ( "?" )*
primary       = literal
              | name ( "." name )*          # each name may carry <...>
              | "{" table "}"
              | "(" params ")" ( "->" type )?
literal       = STRING | NUMBER
table         =                              # empty: {}
              | "[" type "]" ( ":" | "=" ) type sep?     # index signature {[K]: V}
              | name "?"? ( ":" | "=" ) type ( sep name ... )*  # record {a: T, b?: U}
              | type sep?                    # array shorthand {T}
params        = ( "..." type? | (name ":")? type ) ( "," ... )*
sep           = "," | ";"

Atoms in detail

Named types and generic slots <>

A name is an identifier, optionally a dotted path, and each segment may be followed by a generic argument list in angle brackets. The angle-bracket slot is matched by balancing < and > (nesting is allowed), and the language skips whatever is inside — the arguments are not retained. Interpreting them is entirely up to your tool; the bundled language server interprets generic alias instantiation and a small set of type functions such as keyof<T> (see below). The only rule the language enforces is that the brackets balance.

number
Folder
Array<number>
Map<string, Array<number>>   -- nested slots balance
module.Thing<T>                -- dotted path, each segment may slot

Common primitive names a tool will see: boolean number string table thread nil any unknown never void function. These are not keywords to the type grammar — they are just identifiers.

Literal types

A string literal or a numeric literal is a type. These pair naturally with unions.

local mode: "on" | "off" = "on"
local flag: 0 | 1 = 0

Table types {}

Braces cover three shapes, told apart by what is inside. The element and field types may be any type expression, and whitespace and newlines are insignificant, so a brace type can span several lines.

{}                       -- empty table type
{ x: number, y: number }   -- record; fields use : or = , separated by , or ;
{ name?: string }          -- a field may be optional with ?
{ [string]: number }       -- index signature (map from key type to value type)
{ number }                -- array shorthand: a table of number

A single type alone in braces is the array shorthand, and it is valid on its own line. All of these parse:

type This = { C }          -- a table of C
type This = {
  C
}                          -- the same thing across lines
local items: { Widget } = {}   -- and anywhere a type is accepted

A field's type can be any type expression, including a function type, and the parameters of that function may be named. So a record of methods is fine:

type This = {
  test: (self: test) -> ()
}

Function and grouped types ()

Parentheses introduce a parameter list. Followed by -> it is a function type; without an arrow a single type in parentheses is just that type grouped. Parameters may be named (the name is discarded) and the list may end with a vararg ..., optionally typed.

() -> nil
(number, string) -> boolean
(x: number, y: number) -> number   -- parameter names allowed, then ignored
(string, ...number) -> nil          -- trailing vararg, optionally typed
(...) -> any                       -- untyped vararg alone is fine
(...any) -> ...any                  -- a return may be variadic too
(number)                          -- no arrow: just the grouped type number
number -> string                   -- one-parameter sugar: parentheses optional

The single-parameter form without parentheses comes straight from the top-level rule type = union ( "->" type )?: any type followed by an arrow is a one-parameter function type, so T -> R and (T) -> R mean the same thing.

Generic-prefixed function types <T>(...) -> R

A type may open with its own angle-bracket slot before the parameter list. As everywhere else, the language balances and skips the brackets — the prefix declares type parameters for tooling to interpret. This is how a declaration file writes a generic function signature:

<T>(x: T) -> T
<>(...) -> any                     -- an empty slot also parses
<service>(self: any, k: keyof<Services>) -> ValueOf<Services, service>

Type aliases and generics

type Name = T introduces an alias; type Name<A, B> = T introduces a generic alias. The runtime discards the whole statement, but the bundled editor records the alias, substitutes the generic arguments wherever the alias is used, and checks that the right number of arguments is supplied (the GenericArity check):

type Box<T> = { value: T }

local a: Box<number> = { value = 1 }   -- editor sees a as { value: number }
local b: Box = { value = 1 }           -- flagged: type 'Box' expects 1 generic argument(s), got 0

Generic parameter lists on functions and classes (function f<T>(x: T), class C<T>) are accepted in the same skip-the-brackets way. Class, enum, and interface names declared anywhere in the project (or in .luard declaration files) also resolve as types, and a module-qualified name like mod.Thing resolves through the module bound to mod by a require.

Type functions the editor evaluates

Four generic names are not ordinary aliases: the bundled language server computes them when it resolves a type. Each accepts a lowercase or CamelCase spelling. They are still just balanced <...> slots to the runtime — zero effect on execution.

Type functionResult
keyof<T> / KeyOf<T>The union of T's field names as string literal types. T must resolve to a record type.
ValueOf<T, K> / valueof<T, K>The type of field K in record T; K may be a string literal or a union of them (the result is then the union of the matched field types).
NotNil<T> / notnil<T>T with nil removed: unwraps T? and drops nil from unions, following aliases.
ToBasic<T> / tobasic<T>Collapses T to its primitive kind: a literal string type becomes string, true/false become boolean, class instances become class, and so on. A type it cannot simplify is left as written.
type Point = { x: number, name: string }

local k: keyof<Point> = "x"            -- editor resolves the type to "x" | "name"
local n: ValueOf<Point, "x"> = 1       -- editor resolves it to number
local v: ValueOf<Point, keyof<Point>> = 1   -- union of matched fields: number | string

type MaybeName = string | nil
local who: NotNil<MaybeName> = "Ada"   -- editor resolves it to string
local w2: NotNil<string?> = "Bo"       -- same: the ? is unwrapped

local s: ToBasic<"on"> = "off"         -- editor resolves it to string
local t: ToBasic<true> = false         -- editor resolves it to boolean

NotNil checks in the editor

NotNil is more than a resolver: the language server enforces it at three sites, each reported with the [NotNil] tag and silenceable with --#disable NotNil (or --#disable-line / --#disable-next-line):

  • A binding annotated NotNil<...> declared without a non-nil initializer: 'name' is typed NotNil and must be initialized with a non-nil value. Trailing names fed by a multi-value call or ... are excused, since the editor cannot see those values.
  • Assigning the literal nil to such a binding: cannot set 'name' to nil ('name' is typed NotNil).
  • Passing the literal nil in a parameter position typed NotNil: argument 1 to 'greet' cannot be nil (parameter 'who' is typed NotNil).
function greet(who: NotNil<string?>)
  print(`hi {who}`)
end

local name: NotNil<string?> = "Ada"   -- ok
local bad: NotNil<string?>            -- flagged: must be initialized with a non-nil value
name = nil                            -- flagged: cannot set 'name' to nil
greet(nil)                            -- flagged: argument 1 to 'greet' cannot be nil

Remember these are editor diagnostics only — at run time the annotation is gone and nil flows like any other value.

What the runtime does with all of this

Nothing. The interpreter never sees types: they are stripped after parsing. This has a few consequences a tool author should rely on:

  • Annotations can never change behavior, so a checker is free to be as strict or as lenient as it likes without affecting how a program runs.
  • Generic arguments inside <...> are not kept by the language parser at all — if your tool wants them, it must parse the slot itself. (The bundled language server does exactly that: it re-parses the slots to instantiate generic aliases, check arity, and evaluate the type functions above.)
  • A type alias and a :: cast produce no runtime value; they are purely declarations for tooling.
  • Because nothing is enforced at run time, a syntactically valid type that is semantically nonsense will still run. Diagnosing that is the tool's job, not the interpreter's.
