Getting started

This page takes you from a fresh clone of the repository to running scripts, editing with full language support in VS Code, and laying out a small project. A LUAR source file uses the .luar extension.

print("hello, world")

What you need

  • A recent stable Rust toolchain (install with rustup). The crates use the 2024 edition, so you need Rust 1.85 or newer.
  • Node.js and npm — only if you want to build the VS Code extension yourself.

Building from source

The repository holds four independent crates (there is no workspace root, so each one builds on its own):

FolderWhat it is
Luar/The language itself: lexer, parser, interpreter, Ferrite checker, and the Rust host API.
Luar-Runtime/The luar-runtime command-line runner.
Luar-LSP/The luar-lsp language server, with the VS Code extension under editors/code/.
testbed/A small example host that embeds the interpreter.

To get a runner you can call from the command line:

git clone https://github.com/thekingofspace/Luar
cd Luar/Luar-Runtime
cargo build --release

The binary lands at target/release/luar-runtime (luar-runtime.exe on Windows). Put it on your PATH, or run scripts directly through Cargo with cargo run --release -- path/to/script.luar.

Running scripts

luar-runtime hello.luar      # run a source file
luar-runtime game.luarb      # run a precompiled file
luar-runtime game            # bare name: tries game.luar, then game.luarb
luar-runtime my-project/     # directory: tries main.luar, main.luarb, init.luar, init.luarb inside it
luar-runtime                 # no argument: runs main.luar in the current directory

The directory containing the script becomes the base directory for require, so a script can load its neighbours by name. When the script finishes, the runner waits for any coroutines still working in the background before it exits. On failure it prints luar-runtime: <error> and exits with code 1.

File types

ExtensionMeaning
.luarA LUAR source file.
.luarbA precompiled program (a serialized syntax tree). The runner and require load it directly. See Precompilation.
.luardAn ambient declaration file for the editor: its declarations become known globals in every file for hover, completion, and type checking. It is never executed. See Editor & LSP.

A first program

Here is a small but complete script: a Signal class that stores listener functions and fires them all with a value. It shows classes, access control, anonymous functions, and interpolated strings working together. Save it as signal.luar and run it with luar-runtime signal.luar.

class Signal {
  private listeners = {}

  function connect(fn)
    table.insert(self.listeners, fn)
  end

  function fire(value)
    for _, fn in ipairs(self.listeners) do
      fn(value)
    end
  end
}

const onScore = Signal()

onScore:connect(function (points)
  print(`scored {points} points`)
end)

onScore:connect(function (points)
  if points >= 100 then print("new high score!") end
end)

onScore:fire(120)

Running it prints:

scored 120 points
new high score!

A few things worth noticing. Each instance gets its own fresh listeners table, because field defaults are evaluated once per instance at construction. The field is private, so only the class's own methods can touch it. And const makes onScore a binding that cannot be reassigned.

Precompiled scripts

A .luarb file is the same program with lexing and parsing already done. The runner executes it exactly like source, and require falls back to module.luarb when module.luar is missing, so you can ship bytecode-only modules. To produce one, use the host API from Rust:

let bytes = luar::precompile_source(&src)?;   // Vec<u8>, the packed program
std::fs::write("game.luarb", &bytes)?;

See Precompilation for what the format is and when to reach for it.

The VS Code extension

The extension gives you syntax highlighting, diagnostics (Ferrite plus the editor-only checks), hover documentation, completion, inlay type hints, automatic end insertion, and auto-import of project modules. It is built from Luar-LSP/editors/code and packaged as a .vsix file:

# 1. build the language server
cd Luar-LSP
cargo build --release

# 2. place the server binary where the extension looks for it
cp target/release/luar-lsp editors/code/server/      # luar-lsp.exe on Windows

# 3. build and package the extension
cd editors/code
npm install
npm run build
npm run package          # produces luar-lang-<version>.vsix

Install the result with code --install-extension luar-lang-<version>.vsix, or from the Extensions panel via Install from VSIX. The extension activates for .luar and .luard files. If you would rather keep the server binary somewhere else, point the luar.serverPath setting at it; when no server is found, syntax highlighting still works and the language features are disabled. The command Luar: Restart Language Server restarts the server after you rebuild it.

Settings live under luar.*: inlayHints, showMutability, autoIndent, autoImport, serverPath, and trace.server. They are described in Editor & LSP.

Setting up a project

A project is just a folder of .luar files. A typical layout:

my-game/
  main.luar          # entry point: luar-runtime my-game/
  luari.json         # optional: require aliases and editor settings
  lib/
    util.luar
  config/
    settings.luar

Modules load each other with require. A path starting with ./ resolves relative to the requiring file's directory, and the module's returned value is what require hands back:

-- lib/util.luar
function greet(name)
  return `hello, {name}`
end

return { greet = greet }
-- main.luar
local util = require("./lib/util")
print(util.greet("world"))

A luari.json file (the names luari and .luari also work) defines @alias names for require, and can list extra ambient .luard files for the editor:

{
  "aliases": {
    "Settings": "./config/settings"
  },
  "luard": ["./types/globals.luard"]
}

With that in place, any module in the project can write require("@Settings") regardless of where it sits, and require("@Settings/extra") appends a subpath. The runtime searches upward from the requiring module's directory for the config file, so one file at the project root covers everything. The "luard" key (a single path or an array) is read only by the editor. Each module runs once and is cached, and circular requires are reported as errors with the full chain. See Modules for the complete resolution rules.
