Classes

A class is a blueprint for an object. It describes the data an object holds (its fields) and the things it can do (its methods). LUAR is class-based, which means objects are created from classes rather than from raw tables. If you have never used classes before, read this page top to bottom; each idea builds on the last.

Declaring a class and making an object

Declare a class with class Name { ... }. To make an object (also called an instance) you call the class like a function. The values you pass go to the constructor.

class Point {
  public x: number = 0
  public y: number = 0

  constructor(x, y)
    self.x = x
    self.y = y
  end

  function length()
    return math.sqrt(self.x * self.x + self.y * self.y)
  end
}

const p = Point(3, 4)   -- make a Point object
print(p.x)              -- read a field: 3
print(p:length())       -- call a method: 5

self, and the difference between dot and colon

Inside a method, self is the object the method was called on. It is how a method reads and writes that object's own fields.

  • Use a dot (p.x) to read or write a field.
  • Use a colon (p:length()) to call a method. The colon quietly passes the object as self. Writing p.length() would not pass self and is almost never what you want.

Fields

A field is a named piece of data on every instance. A field may have a type annotation (for the editor) and a default value that is set when the object is built.

class Player {
  public name: string = "anon"
  public level = 1             -- the type annotation is optional
  public inventory: table = {}   -- each instance gets its own table
}

Access control: public, protected, private

Access control limits who is allowed to touch a member. It helps you keep the inside of an object separate from the outside.

ModifierReachable from
public (default)Anywhere.
protectedThe declaring class and its subclasses only.
privateThe declaring class's own methods only.
class Account {
  public owner: string = ""
  private balance: number = 0     -- outsiders cannot read or write this

  function deposit(amount)
    self.balance = self.balance + amount   -- allowed: same class
  end
  function total() return self.balance end
}

Reading or writing a member you are not allowed to touch is a run-time error. This is enforced while the program runs, not just by the editor.

Static members

A static field or method belongs to the class itself rather than to any one instance. There is exactly one copy, shared by everyone, and you reach it through the class name. This is useful for counters, caches, or factory helpers.

class Counter {
  static total: number = 0
  constructor() Counter.total += 1 end
  static function made() return Counter.total end
}
Counter()  Counter()
print(Counter.made())   -- 2

The constructor

The constructor runs once, when the class is called to build an object. Its job is to set up the new instance, which is available as self. Field defaults are applied first, then the constructor runs. A class without a constructor can still be built; you just get the defaults.

class Box {
  public items: table = {}
  constructor(first)
    table.insert(self.items, first)
  end
}

Methods and their modifiers

A method is a function declared inside the class body. Besides the access words, a method can be marked:

ModifierMeaning
staticBelongs to the class; called on the class name, has no self.
abstractHas no body; a concrete subclass must provide one. See abstract classes.
finalCannot be overridden by a subclass.
overrideStates that this method replaces an inherited one. It is an error if there is nothing to override, which catches typos.

Getters and setters

A getter is a method that runs when you read a property; a setter runs when you assign to it. From the outside they look exactly like a field (no parentheses), but they let you compute a value or validate an assignment.

class Temperature {
  private celsius: number = 0

  get fahrenheit()
    return self.celsius * 9 / 5 + 32
  end
  set fahrenheit(value)
    self.celsius = (value - 32) * 5 / 9
  end
}

const t = Temperature()
t.fahrenheit = 212      -- runs the setter
print(t.fahrenheit)    -- runs the getter: 212

Operator overloading

Operator overloading lets your objects work with the normal operators like + and ==. You write operator SYMBOL(other) and return the result. This is handy for math-like types such as vectors or money.

class Vec {
  public x: number = 0
  public y: number = 0
  constructor(x, y) self.x = x  self.y = y end

  operator +(o)  return Vec(self.x + o.x, self.y + o.y) end
  operator -(o)  return Vec(self.x - o.x, self.y - o.y) end
  operator -()   return Vec(-self.x, -self.y) end      -- unary minus
  operator ==(o) return self.x == o.x and self.y == o.y end
  operator #()   return 2 end                        -- length, used by #v
  operator tostring() return `Vec({self.x}, {self.y})` end
}

print(Vec(1, 2) + Vec(3, 4))   -- Vec(4, 6)

The operators you can overload: +, - (binary and unary), *, /, %, ^, .. (concatenation), ==, <, <=, # (length), and tostring (used by print and tostring).

Inheritance with extends

Inheritance lets one class build on another. The new class (the subclass) gets all the fields and methods of the parent, and can add new ones or replace existing ones. A class extends exactly one parent.

class Animal {
  public name: string = "animal"
  constructor(name) self.name = name end
  function describe() return self.name end
  function speak() return "..." end
}

class Dog extends Animal {
  public breed: string = "mutt"
  override function speak() return "woof" end
}

super: reaching the parent

super lets a subclass call the parent's version of something. The two common uses are running the parent constructor and extending an overridden method instead of fully replacing it.

class Dog extends Animal {
  public breed: string = "mutt"
  constructor(name, breed)
    super:constructor(name)      -- run Animal's constructor first
    self.breed = breed
  end
  override function describe()
    return super:describe() .. " the " .. self.breed   -- add to the parent's result
  end
}

Abstract classes and methods

An abstract class is one that is not complete on its own and cannot be turned into an object directly; it exists to be extended. An abstract method has no body and is a promise that every concrete subclass will provide one. This is how you define a shared shape that each subclass fills in.

abstract class Shape {
  abstract function area() end     -- no body; subclasses must implement it
  function describe() return `area is {self:area()}` end
}

class Square extends Shape {
  public side: number = 0
  override function area() return self.side * self.side end
}

-- Shape() is an error: you cannot instantiate an abstract class

Final classes and methods

final locks something down. A final class cannot be extended, and a final method cannot be overridden. Use it when a type or behavior is not meant to be changed by subclasses.

final class Money { }
-- class Cash extends Money { }   would be an error

Mixins: reuse without inheritance

A mixin lets you pull methods (and operators and fields) from other classes into yours, without making them your parent. It is a way to share behavior between unrelated classes. Use mixin A, B to mix in one or more classes.

class Loggable {
  function log(msg) print(`[{self.name}] {msg}`) end
}

class Service extends Base mixin Loggable {
  public name: string = "svc"
}

Service():log("started")   -- log came from the mixin

Your own members take priority over mixed-in ones with the same name. The full clause order on a class header is extends Parent mixin A, B implements I, J.

Interfaces

An interface is a contract: a list of member names a class must provide. A class promises to satisfy an interface with implements, and the promise is checked when the class is built (a missing member is an error). Interfaces are about shape, not shared code; they do not provide implementations.

interface Named {
  name
  describe
}

class Tag implements Named {
  public name: string = "tag"
  function describe() return self.name end
}

An interface can build on others with extends, and a class can implement several interfaces at once.

Reflection: inspecting classes at run time

These built-in functions let a running program ask questions about classes and objects.

FunctionResult
instanceof(value, Class)True if the value is an instance of the class or one of its subclasses, or implements the interface.
classname(valueOrClass)The class name as a string.
classof(value)The class object an instance was made from.
superclass(valueOrClass)The parent class, or nil if there is none.
methodsof(valueOrClass)An array of the method names, including inherited ones.
isabstract(valueOrClass)Whether the class is abstract.
print(classname(rex))                  -- Dog
print(classname(superclass(rex)))      -- Animal
print(instanceof(rex, Animal))         -- true
print(isabstract(Shape))               -- true

How an object is built, step by step

When you call a class to make an object, this is the order of events:

  1. A fresh empty instance is created and linked to its class.
  2. Inherited field defaults are applied, from the root parent down to this class.
  3. This class's own field defaults are applied.
  4. The constructor runs (yours, or the inherited one) with the new object as self.
  5. The finished object is returned.

Why classes instead of raw tables and metatables

Under the hood, objects are tables with a metatable, and you can build objects by hand with setmetatable and __index (see Metatables). For most code, a class is the better choice, because it gives you the same machinery with far less ceremony and far more help.

  • Less boilerplate. Inheritance, the constructor, method lookup, and the back-link from an instance to its type are all set up for you. Doing the same with metatables means wiring up __index chains and copying defaults by hand, which is repetitive and easy to get subtly wrong.
  • Real access control. Classes enforce private and protected at run time. Plain tables expose every field to everyone, so there is nothing stopping outside code from reaching in and breaking your object's assumptions.
  • Safety checks. The language verifies that override really overrides something, that abstract methods get implemented, that a final class is not extended, and that a class actually implements the interfaces it claims. A hand-rolled metatable gives you none of these.
  • The editor understands it. Because a class is a declared shape, the editor can complete its fields and methods, show its types on hover, and follow inheritance and mixins. A raw table assembled at run time is mostly opaque to tooling.
  • Clear intent. class Dog extends Animal says exactly what you mean. A metatable with an __index pointing at another table is a pattern a reader has to decode.

Use metatables directly when you need a behavior a class does not model, such as a custom __index function that computes values on demand, a __call handler, or a proxy. For ordinary objects with fields, methods, and inheritance, use a class.
