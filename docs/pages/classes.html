Classes

A class is a blueprint for an object. It describes the data an object holds (its fields) and the things it can do (its methods). LUAR is class-based, which means objects are created from classes rather than from raw tables. If you have never used classes before, read this page top to bottom; each idea builds on the last.

Declaring a class and making an object

Declare a class with class Name { ... }. To make an object (also called an instance) you call the class like a function. The values you pass go to the constructor.

class Point {
  public x: number = 0
  public y: number = 0

  constructor(x, y)
    self.x = x
    self.y = y
  end

  function length()
    return math.sqrt(self.x * self.x + self.y * self.y)
  end
}

const p = Point(3, 4)   -- make a Point object
print(p.x)              -- read a field: 3
print(p:length())       -- call a method: 5

The full class header is:

[pub] [final] [abstract] class Name [extends Parent] [mixin A, B] [implements I, J] { ... }

The leading modifiers (pub, final, abstract) may appear in any order — abstract pub class and pub abstract class mean the same thing. The clauses after the name, however, are positional: extends first, then mixin, then implements. A class name may also carry a generic parameter list (class Box<T>); the brackets are read and discarded by the runtime and used only by the editor (see Type rules).

self, and the difference between dot and colon

Inside a method, self is the object the method was called on. It is how a method reads and writes that object's own fields.

  • Use a dot (p.x) to read or write a field.
  • Use a colon (p:length()) to call a method. The colon quietly passes the object as self. Writing p.length() would not pass self and is almost never what you want.

Fields

A field is a named piece of data on every instance. A field may have a type annotation (for the editor) and a default value that is set when the object is built.

class Player {
  public name: string = "anon"
  public level = 1             -- the type annotation is optional
  public inventory: table = {}   -- each instance gets its own table
}

Access control: public, protected, private

Access control limits who is allowed to touch a member. It helps you keep the inside of an object separate from the outside. The check is made against the class whose code is currently running (a method body, a constructor, a field default — see also the closure rule below).

ModifierReachable from
public (default)Anywhere.
protectedCode running in the declaring class or any class that descends from it.
privateCode running in the exact declaring class only — not even its subclasses.

Access words apply to fields, methods, getters, and setters alike, and an inherited member keeps the access level of the class that declared it.

class Account {
  public owner: string = ""
  private balance: number = 0     -- outsiders cannot read or write this

  function deposit(amount)
    self.balance = self.balance + amount   -- allowed: same class
  end
  function total() return self.balance end
}

const a = Account()
a:deposit(50)
print(a:total())   -- 50
print(a.balance)   -- runtime error: member 'balance' is private to class 'Account'

Reading or writing a member you are not allowed to touch is a run-time error of the form member 'balance' is private to class 'Account' (or ... is protected to class '...'). This is enforced while the program runs, not just by the editor.

Closures keep the access rights of where they were written

Access is lexical: any function literal created inside a class body — in a method, a constructor, a getter, or a field default — permanently remembers that class, and runs with that class's access rights every time it is called, even long after the method that made it has returned. A closure created outside the class never gets those rights, no matter which objects it captures.

class Tally {
  private count: number = 0

  function Bumper()
    -- written inside a method, so it keeps Tally's private access forever
    return function()
      self.count = self.count + 1
      return self.count
    end
  end
}

const bump = Tally():Bumper()
bump()  bump()
print(bump())   -- 3: the closure may touch the private field

const t = Tally()
const peek = function() return t.count end   -- written outside the class
peek()   -- runtime error: member 'count' is private to class 'Tally'

This means a class can hand out workers that operate on its own internals without opening those internals to everyone. When a closure is invoked as a method of some class, the method's own class is used for the check instead.

The encapsulation pattern: expose an operation, not a field

Because private means "the exact declaring class only", one class can never reach into another class's private data — not even a closely related one. The right design is for the owning class to expose a small public method that performs the operation, so the data structure itself stays private. A classic example is a signal whose connections can remove themselves:

class Connection {
  private signal = nil
  private fn = nil
  constructor(signal, fn)
    self.signal = signal
    self.fn = fn
  end
  function Call(msg) self.fn(msg) end
  function Disconnect()
    -- ask the signal to remove us; we cannot touch its private list
    self.signal:Remove(self)
  end
}

class Signal {
  private listeners: table = {}

  function Connect(fn)
    local c = Connection(self, fn)
    table.insert(self.listeners, c)
    return c
  end

  -- the public, sanctioned way to mutate the private list
  function Remove(connection)
    for i, c in ipairs(self.listeners) do
      if c == connection then
        table.remove(self.listeners, i)
        return
      end
    end
  end

  function Fire(msg)
    for _, c in ipairs(self.listeners) do
      c:Call(msg)
    end
  end
}

const s = Signal()
const conn = s:Connect(function(msg) print(`got {msg}`) end)
s:Fire("hello")     -- got hello
conn:Disconnect()   -- Connection asks; Signal does the removing
s:Fire("again")     -- nothing: the listener is gone

If Disconnect tried self.signal.listeners directly it would fail with member 'listeners' is private to class 'Signal'. Exposing Remove(connection) keeps the invariant (only Signal edits its list) while still letting other objects request the operation. Note the comparison c == connection: instances compare by identity, so this finds exactly the connection that was handed out (see instance equality).

Static members

A static field or method belongs to the class itself rather than to any one instance. There is exactly one copy, shared by everyone, and you reach it through the class name. This is useful for counters, caches, or factory helpers.

class Counter {
  static total: number = 0
  constructor() Counter.total += 1 end
  static function made() return Counter.total end
}
Counter()  Counter()
print(Counter.made())   -- 2

The constructor

The constructor runs once, when the class is called to build an object. Its job is to set up the new instance, which is available as self. Field defaults are applied first, then the constructor runs. A class without a constructor can still be built; you just get the defaults.

class Box {
  public items: table = {}
  constructor(first)
    table.insert(self.items, first)
  end
}

Constructors are inherited: if a class declares no constructor of its own, the nearest ancestor's constructor runs instead, so a subclass that adds nothing to construction does not need to repeat it. A subclass that does declare one can run the parent's explicitly with super:constructor(...) (see super).

The destructor

A destructor is the mirror of a constructor: it runs once, automatically, the moment an instance is about to be released. Release happens deterministically when the last binding that holds the object disappears — it leaves its scope (the end of a do block, a function return), it is reassigned to a new value, or it is set to nil — and nothing else refers to it. Use it for cleanup — closing a handle, removing a drawn element, releasing a resource.

class Handle {
  public id: number = 0
  constructor(id) self.id = id end
  destructor() print(`closing {self.id}`) end
}

do
  local h = Handle(1)   -- used here
end                     -- destructor runs now: "closing 1"

An object that is still referenced elsewhere — returned from a function, stored in a table, captured by a closure — is not released, so its destructor does not run until that last reference also goes away. This makes cleanup predictable: it happens exactly when the object is truly no longer in use.

Destructors are not inherited. When a subclass needs cleanup it must declare its own destructor — a parent's destructor does not run for a subclass instance that lacks one. Each instance runs at most the destructor declared on its own class.

Methods and their modifiers

A method is a function declared inside the class body. Besides the access words (public, protected, private), a method can be marked:

ModifierMeaning
staticBelongs to the class; called on the class name, has no self.
abstractHas no body; a concrete subclass must provide one. See abstract classes.
finalCannot be overridden by a subclass.
overrideStates that this method replaces an inherited one. If nothing is actually overridden the class fails to load with method 'name' is marked `override` but does not override anything, which catches typos.

Member modifiers may be combined and written in any order: static public function f() and public static function f() are identical, and the same goes for every combination of private | protected | public | static | abstract | final | override.

Getters and setters

A getter is a method that runs when you read a property; a setter runs when you assign to it. From the outside they look exactly like a field (no parentheses), but they let you compute a value or validate an assignment.

class Temperature {
  private celsius: number = 0

  get fahrenheit()
    return self.celsius * 9 / 5 + 32
  end
  set fahrenheit(value)
    self.celsius = (value - 32) * 5 / 9
  end
}

const t = Temperature()
t.fahrenheit = 212      -- runs the setter
print(t.fahrenheit)    -- runs the getter: 212
t.fahrenheit += 10      -- compound assignment reads via the getter, writes via the setter

Getters and setters also intercept compound assignment (+=, -=, *=, /=, %=, ..=): the current value is read through the getter, combined, and written back through the setter. They take access modifiers like any member, and they are inherited by subclasses.

A getter or setter fires on every access of that name, including self.name inside the class itself. Store the real data in a differently-named (usually private) field, as Temperature stores celsius behind the fahrenheit accessors — a setter that assigns to its own name would recurse forever.

Operator overloading

Operator overloading lets your objects work with the normal operators like + and ==. You write operator SYMBOL(other) and return the result. This is handy for math-like types such as vectors or money.

class Vec {
  public x: number = 0
  public y: number = 0
  constructor(x, y) self.x = x  self.y = y end

  operator +(o)  return Vec(self.x + o.x, self.y + o.y) end
  operator -(o)  return Vec(self.x - o.x, self.y - o.y) end
  operator -()   return Vec(-self.x, -self.y) end      -- unary minus
  operator ==(o) return self.x == o.x and self.y == o.y end
  operator #()   return 2 end                        -- length, used by #v
  operator tostring() return `Vec({self.x}, {self.y})` end
}

print(Vec(1, 2) + Vec(3, 4))   -- Vec(4, 6)

The operators you can overload: +, - (binary and unary — an operator - with zero parameters is unary minus), *, /, %, ^, .. (concatenation), ==, <, <=, # (length), tostring (used by print and tostring), and type (used by TypeOf, below). Declaring any other symbol fails with unsupported operator overload '...'.

Some comparisons are derived rather than declared:

  • a ~= b is the negation of your operator ==.
  • a > b runs operator < with the operands swapped, and a >= b swaps operator <= — so defining < and <= gives you all four orderings.

A binary operator fires when either operand is an instance of a class that defines it (the instance side becomes self, the other operand is the parameter). There is no call operator: a class instance cannot be made callable with obj(...). The __call metamethod exists only for plain tables built with setmetatable (see Metatables).

The type operator and TypeOf

operator type gives your class a custom deep type name, returned by the built-in TypeOf for both instances and the class value itself. It must return a string. The plain type() function is not affected — an instance is still a table.

class Vec {
  operator type() return "Vec" end
}

print(TypeOf(Vec()))   -- Vec
print(TypeOf(Vec))     -- Vec (works on the class value too)
print(type(Vec()))     -- table (type() ignores the overload)

Inheriting and overriding operators

Operators are inherited just like methods: a subclass automatically gets the operators of its parent (and of any mixins), and can replace one by declaring the same operator again. super only dispatches method calls, so a subclass operator cannot invoke the parent's operator directly — when parent and child should share logic, put it in an ordinary method and reach that through super:

class Money {
  public cents: number = 0
  constructor(c) self.cents = c end
  function addCents(o) return self.cents + o.cents end
  operator +(o) return Money(self:addCents(o)) end
}

class Wallet extends Money {
  operator +(o) return Wallet(super:addCents(o)) end   -- shared logic via a method
}

When operators come from more than one place, the lookup order is the same as for methods: the class's own operator wins over a mixed-in one, which wins over an inherited one.

Inheritance with extends

Inheritance lets one class build on another. The new class (the subclass) gets all the fields and methods of the parent, and can add new ones or replace existing ones. A class extends exactly one parent.

class Animal {
  public name: string = "animal"
  constructor(name) self.name = name end
  function describe() return self.name end
  function speak() return "..." end
}

class Dog extends Animal {
  public breed: string = "mutt"
  override function speak() return "woof" end
}

The parent is looked up by name when the class statement runs, so it must already be defined. Extending a name that does not exist fails with unknown parent class 'X', and extending something that is not a class fails with 'X' is not a class.

super: reaching the parent

super lets a subclass call the parent's version of something. The two common uses are running the parent constructor and extending an overridden method instead of fully replacing it.

class Dog extends Animal {
  public breed: string = "mutt"
  constructor(name, breed)
    super:constructor(name)      -- run Animal's constructor first
    self.breed = breed
  end
  override function describe()
    return super:describe() .. " the " .. self.breed   -- add to the parent's result
  end
}

Inside a method or constructor, self and super are injected automatically as constants. super is the parent class value, and super:method(...) dispatches the parent's implementation with the current self — that is what makes "extend instead of replace" work. super supports method calls and super:constructor(...) only; there is no super form for fields or operators.

Abstract classes and methods

An abstract class is one that is not complete on its own and cannot be turned into an object directly; it exists to be extended. An abstract method has no body and is a promise that every concrete subclass will provide one. This is how you define a shared shape that each subclass fills in.

abstract class Shape {
  abstract function area() end     -- no body; subclasses must implement it
  function describe() return `area is {self:area()}` end
}

class Square extends Shape {
  public side: number = 0
  override function area() return self.side * self.side end
}

-- Shape() is an error: you cannot instantiate an abstract class

The rules, with their exact errors:

  • Calling an abstract class fails immediately: cannot instantiate abstract class 'Shape'. Its subclasses are the things you instantiate.
  • An abstract method must have an empty body: just abstract function area() end. The body of an abstract method can never run, and the linter flags a non-empty one with abstract method 'area' must not have a body (AbstractMethodWithBody).
  • If a concrete subclass forgets to implement an abstract method, the mistake surfaces when the method is called: abstract method 'area' has no implementation in class 'Shape'. The same error fires from self:area() inside an inherited method like describe above.
  • A method cannot be both final and abstract — the linter reports method 'm' cannot be both `abstract` and `final` (FinalAndAbstract).

An abstract class may freely mix abstract and concrete members; only the abstract ones impose obligations on subclasses.

Final classes and methods

final locks something down. A final class cannot be extended, and a final method cannot be overridden. Use it when a type or behavior is not meant to be changed by subclasses.

final class Token {
  public id: number = 0
}
-- class FakeToken extends Token {}   -- error: cannot extend final class 'Token'

class Base {
  final function checksum() return 42 end
}
class Sub extends Base {
  -- function checksum() ... end     -- error: cannot override final method 'checksum'
}

Both checks happen at run time, when the offending class statement executes:

  • Extending a final class fails with cannot extend final class 'Token'.
  • Declaring a method whose name is final anywhere up the parent chain fails with cannot override final method 'checksum' — with or without the override keyword.

The editor catches both ahead of time with the FinalOverride check (an error-severity diagnostic): cannot extend final class 'Token' — this errors at runtime and cannot override final method 'checksum' from class 'Base' — this errors at runtime.

Mixins: reuse without inheritance

A mixin lets you pull methods (and operators and fields) from other classes into yours, without making them your parent. It is a way to share behavior between unrelated classes. Use mixin A, B to mix in one or more classes.

class Loggable {
  function log(msg) print(`[{self.name}] {msg}`) end
}

class Service extends Base mixin Loggable {
  public name: string = "svc"
}

Service():log("started")   -- log came from the mixin

A mixin brings in the other class's methods, operator overloads, and fields (with their default values). When the same name comes from several places, resolution follows a clear order:

  • A member declared in the class itself always wins.
  • Otherwise a mixed-in method or operator wins over an inherited (extends) one.
  • If two mixins define the same method or operator, the last one in the mixin list wins.
  • Fields work the other way around: a mixin's field is only added when no field of that name exists yet, so the parent's default (or an earlier mixin's) is kept. A field declared in the class body itself still replaces any inherited or mixed-in default.

Naming a mixin that does not exist fails with unknown mixin class 'X', and mixing in a non-class fails with 'X' is not a class (mixin). The full clause order on a class header is always extends Parent mixin A, B implements I, J. A class still has exactly one parent; mixins are how you compose behavior from several classes at once.

Interfaces

An interface is a contract: a list of member names a class must provide. A class promises to satisfy an interface with implements, and the promise is checked when the class is built (a missing member is an error). Interfaces are about shape, not shared code; they do not provide implementations.

interface Named {
  name
  describe
}

class Tag implements Named {
  public name: string = "tag"
  function describe() return self.name end
}

Inside an interface body each line names one required member. For readability you may also write the function keyword, a parameter list, and a type annotation — all are accepted and discarded; only the names are kept:

interface Named {
  name: string
  function describe(): string
}

How conformance is checked:

  • The check runs once, when the class statement executes. A missing member fails with class 'Tag' does not implement member 'describe' required by interface 'Named'.
  • A member may be satisfied by a method, a field, a getter or setter, or a static — anything with that name, on the class or inherited from a parent.
  • An interface can build on others with extends I, J; a class must then satisfy the inherited members too.
  • A class can implement several interfaces at once: implements Named, Serializable. Naming something that is not an interface fails with 'X' is not an interface (or unknown interface 'X' if undefined).
  • instanceof(value, SomeInterface) returns true when the value's class (or any ancestor) implements the interface or one that extends it.

Interfaces are about shape only — they carry no code and no defaults. If you want to share an implementation, use a mixin instead.

Instance equality

By default, == on two instances compares identity: it is true only when both sides are the very same object. Two instances built from the same values are still different objects. This is what makes patterns like the Signal's Remove(connection) search reliable.

class Point {
  public x = 0
  constructor(x) self.x = x end
}

const a = Point(1)
const b = Point(1)
print(a == b)   -- false: different objects
print(a == a)   -- true: same object

Declaring operator == replaces this with value comparison for that class (see operator overloading above); the overload fires when either operand is an instance of a class that defines it, and ~= is always its negation. rawequal(a, b) ignores overloads and always compares identity.

Reflection: inspecting classes at run time

These built-in functions let a running program ask questions about classes and objects.

FunctionResult
instanceof(value, Class)True if the value is an instance of the class or one of its subclasses, or implements the interface.
classname(valueOrClass)The class name as a string.
classof(value)The class object an instance was made from.
superclass(valueOrClass)The parent class, or nil if there is none.
methodsof(valueOrClass)An array of the method names, including inherited ones.
isabstract(valueOrClass)Whether the class is abstract.
print(classname(rex))                  -- Dog
print(classname(superclass(rex)))      -- Animal
print(instanceof(rex, Animal))         -- true
print(isabstract(Shape))               -- true

How an object is built, step by step

When you call a class to make an object, this is the order of events:

  1. If the class is abstract, the call fails right here.
  2. A fresh empty instance is created and linked to its class.
  3. Field defaults are applied in order: the parent chain's first, then any mixins', then this class's own (an own field replaces an inherited default of the same name). Defaults are evaluated per instance, with self already in scope, so {} gives every instance its own table.
  4. The constructor runs — this class's own, or the nearest ancestor's if this class declares none — with the new object as self.
  5. The finished object is returned.

Why classes instead of raw tables and metatables

Under the hood, objects are tables with a metatable, and you can build objects by hand with setmetatable and __index (see Metatables). For most code, a class is the better choice, because it gives you the same machinery with far less ceremony and far more help.

  • Less boilerplate. Inheritance, the constructor, method lookup, and the back-link from an instance to its type are all set up for you. Doing the same with metatables means wiring up __index chains and copying defaults by hand, which is repetitive and easy to get subtly wrong.
  • Real access control. Classes enforce private and protected at run time. Plain tables expose every field to everyone, so there is nothing stopping outside code from reaching in and breaking your object's assumptions.
  • Safety checks. The language verifies that override really overrides something, that abstract methods get implemented, that a final class is not extended, and that a class actually implements the interfaces it claims. A hand-rolled metatable gives you none of these.
  • The editor understands it. Because a class is a declared shape, the editor can complete its fields and methods, show its types on hover, and follow inheritance and mixins. A raw table assembled at run time is mostly opaque to tooling.
  • Clear intent. class Dog extends Animal says exactly what you mean. A metatable with an __index pointing at another table is a pattern a reader has to decode.

Use metatables directly when you need a behavior a class does not model, such as a custom __index function that computes values on demand, a __call handler, or a proxy. For ordinary objects with fields, methods, and inheritance, use a class.
