Garbage collection

LUAR manages memory automatically. Heap objects (tables, functions, and classes) are reference-counted, so most values are reclaimed the instant the last reference to them disappears. On top of that, a tracing mark-and-sweep collector runs periodically to reclaim reference cycles — structures that point at each other and would otherwise keep themselves alive forever. You never free memory by hand; at most you drop a reference and let the system do the rest.

local a = {}
local b = {}
a.other = b
b.other = a          -- a cycle: each table keeps the other alive
a = nil
b = nil              -- unreachable now; the collector reclaims the pair

How a collection works

A collection starts from the roots: every variable on the active scope chain (which ends at the shared global scope), every variable in the module's root scope, and every buff variable. The mark phase follows everything reachable from those roots — table entries, table keys, metatables, the scopes captured by functions, and class members. The sweep phase then empties every table it could not reach. Emptying a table breaks any cycle running through it, and reference counting immediately frees everything the cycle was holding.

Collections run at safe points between top-level statements — the interpreter never interrupts a statement halfway through. A collection is scheduled automatically once enough allocations have happened since the last one (10,000 by default), or on request with collectgarbage().

collectgarbage()

The collectgarbage() builtin requests a collection: it flags the collector, and the actual collection runs at the next safe point. It takes no arguments and returns nothing.

build_temporary_world()      -- allocates heavily, then drops it all
collectgarbage()             -- reclaim it at the next statement boundary

Freeing values yourself

You rarely need to. Assigning nil to a binding drops its reference, and the value is reclaimed once nothing else holds it. This is allowed even on an immutable binding — setting a const to nil is the sanctioned way to "free" it (any other reassignment is still an error).

const cache = load_cache()
-- ... use it ...
cache = nil   -- legal on a const: releases the binding

Two kinds of storage opt out of automatic collection: buff variables are roots that never go out of scope and are released only with freebuff, and a required module stays cached for the life of the program (a Rust host can evict one — see below).

Destructors run deterministically

A class destructor does not wait for the collector. It runs the moment the last variable holding the instance lets go of it — provided nothing else still refers to the object at that moment:

EventExample
The scope holding the only reference endsthe end of a do block, a function returning, a loop iteration finishing
The binding is reassignedh = other
The binding is set to nilh = nil
A new declaration replaces the bindinglocal h = Handle(2) over an existing h

If other references are still alive at that point (the instance was returned, stored in a table, or captured by a closure), the destructor is skipped and the object lives on. A destructor never re-enters itself, so cleanup code cannot trigger the same destructor recursively. See Classes for how to declare one.

class Handle {
  public id: number = 0
  constructor(id) self.id = id end
  destructor() print(`closing {self.id}`) end
}

do
  local h = Handle(1)
end                     -- prints "closing 1" right here

One ordering rule worth knowing: a tail call (return f(x) as the entire return) dismantles the calling frame before f runs, so destructors for the caller's locals fire before the tail-called function executes, not after it returns.

local function step()
  local h = Handle(7)
  return finish()      -- tail call: "closing 7" prints before finish() runs
end

Script ownership (for Rust hosts)

Every function, table, and class remembers the script that created it. A script is one top-level program or one required module, identified by a numeric id. This lets a host that loads and unloads scripts at runtime tear one down completely without restarting the interpreter:

Interpreter methodWhat it does
current_script_id()The id of this interpreter's own script.
script_of_value(&v)The id of the script that created a function, table, or class value.
source_of_value(&v)The source file path of that script, when one was registered with set_source_path.
has_live_functions(id) / live_function_count(id)Whether (and how many) functions created by the script are still reachable.
free_script(id)Kills every function the script created — marks them dead and replaces them with nil in every scope and every table — then clears the script's root scope.
free_module(name)Resolves name like require would, evicts it from the module cache, and frees its script.
collect_garbage()Runs a mark-and-sweep collection immediately.

Calling a function from a freed script raises attempt to call a function from a freed script. This is the defensive backstop: after free_script the references themselves have already been nilled out everywhere, so a stale call can only happen through a handle the host itself kept.

Lower-level knobs live in luar::runtime::gc: set_threshold(n) changes how many allocations pass between automatic collections, request() and should_collect() schedule and inspect a pending collection, and live_objects() counts the live tables and functions on the current thread's heap. The heap is per-thread: an interpreter and every value it creates must stay on the thread that created them.
