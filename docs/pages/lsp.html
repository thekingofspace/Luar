Editor & LSP

LUAR ships with a language server and a VS Code extension (luar-lsp, in Luar-LSP/). The server resolves the type of every binding by walking the real parser's AST, honours every written annotation, and powers completion, hover, inlay hints, semantic coloring, and diagnostics. This page explains what the tooling does, the built-in type functions, and .luard ambient files.

A manually written type always wins over an inferred one. Anywhere you write : T or cast with :: T, the tool takes your word for it.

What gets resolved

Every value form the language has: literals, table constructors, class instantiation (Point(3, 4) → Point), enum members, function signatures with per-position return unions, switch expressions, and/or operand unions, operator overloads resolved by symbol and arity, getters before fields along the whole parent chain, parent fields before mixin fields, and conditional assignments unioned at the join point. Reassigning a const or bare binding to anything but nil is flagged as an error, matching Ferrite.

Completion

Completion triggers on . : " ' @ / and #, and on plain identifiers.

  • Names in scope — every binding declared at or above the cursor, the parameters of the enclosing function / constructor / operator / get / set block, and loop variables. Parameters are found by a textual block scan, so they complete even while the file is mid-edit and does not parse. Globals from .luard files, pub globals from other files, and the builtin library round out the list; function names insert as call snippets.
  • Members — v. lists fields and getters; v: lists methods. Inheritance, mixins, interfaces, and override are followed. Private and protected members are hidden outside their class; self. sees everything.
  • Chains — calls resolve through their return types: b:with(x):with(y). works, including builder types whose methods return the same type, and (value :: T).field casts.
  • Types — after :, ::, in type X = ... bodies, and in return positions. Generic parameters in scope (function f<T>, type Box<T>) are offered too, along with the type functions below as snippets.
  • Module types — in a type position, Mod. lists the export type names of the module bound to Mod by a require.
  • Literal unions — a binding typed "on" | "off" (directly, through an alias, or through keyof) offers its strings when assigning, comparing, in case patterns, and as call arguments (go(" suggests the parameter's literals).
  • Require paths — files and folders, @aliases, and @self (only inside init.luar). The requiring file is never listed for ./.
  • Bracket keys — t[" offers the table's string keys; t[ offers them too and accepting one inserts "key"]. Keys that cannot be written with a dot (such as ["easter.Test tes"]) also appear in t. completion as bracketed entries — accepting one removes the dot and writes the bracket form. t: completion drops bracket-only keys.
  • Auto-import — project modules not yet required, sorted to the bottom of the list (see the luar.autoImport setting).
  • Directives — after --# the editor completes disable / disable-line / disable-next-line, then the check names (see Ferrite lints in the editor below).

Hover and documentation comments

Hovering a function shows its full signature (clamp(x: number, lo: number) -> number); other bindings show their declaration form (const name: T, local name: T, buff name: T, class Name, enum Name, interface Name). Hovering self or super shows the enclosing class or its parent.

A block of consecutive -- comment lines written directly above a declaration is shown as documentation on hover — for class members too, following the parent chain to find the documented definition, and for .luard globals, reading the declaring ambient file. Extra leading dashes and --[[ ]] markers are stripped, so --- doc style works. A --# directive line is not documentation and ends the block.

-- Clamps x into the inclusive range lo..hi.
-- Returns lo when x is below it, hi when above.
function clamp(x: number, lo: number, hi: number): number
  return math.clamp(x, lo, hi)
end

Inlay hints

Each declaration line gets a hint with the binding's resolved type, rendered right after the name — writing local hp = 100 displays as if it were local hp: mut number = 100. The rules:

  • Hints appear on declarations (local, const, pub), on bare assignments (shown as imut — they declare an immutable binding), and on buff declarations (shown as mut). Reassignments get no hint.
  • A binding with a written : T annotation is skipped — you already said what it is. A :: T cast is not an annotation, so the resolved type still shows.
  • Lines declaring a function (function name(...)) are skipped.
  • With luar.showMutability off the label is just : type. Labels longer than 48 characters are truncated with ….

Auto-end insertion

With luar.autoIndent on, pressing Enter runs two checks against a copy of the document with comments and strings stripped (block comments, long strings, and backtick strings are all understood):

  • Missing keyword — if the line you left is an if / elseif header without then, or a for / while header without do, the keyword is appended to the line. This only happens when the header actually has a condition — the line must end in a word, closing bracket, or quote, and not in a trailing and/or/not.
  • Missing end — if the line opens a block (if, for, while, function, switch, case, default, constructor, destructor, operator, get name( / set name(, or a leading do) and the whole document has more openers than ends, an end is inserted on the line below the cursor at the opener's indentation. elseif never gets an end. If the next line holds only closers ()]},;), the end goes in front of them.

The balance counting knows that for ... do and while ... do are one block, not two, and that interface { ... } bodies are closed by braces rather than end.

Semantic coloring of .luard globals

Identifiers that name a global declared in a .luard ambient file are emitted as semantic tokens (function for function-typed globals, variable otherwise, always with the defaultLibrary modifier), so an engine's injected API is visibly distinct everywhere it is used. The extension maps both to the support.function.builtin.luar TextMate scope, which themes color like builtins. Identifiers directly after . or : are skipped, so fields that merely share a name stay uncolored. The extension also ships default colors for enum names (entity.name.type.enum.luar) and enum members (variable.other.enummember.luar).

Diagnostics

These are produced by the language server itself; Ferrite's lints are merged in alongside them (see below). Every code here honours the same --#disable directives as Ferrite.

Modules in the editor

The resolver mirrors the runtime: ./ is the file's directory and each extra dot climbs one level (../, .../, …); a folder with init.luar resolves to it, a folder without one becomes a table of its modules; require takes the type of whatever the module returns. Inside init.luar, ./ resolves to the folder's parent — the file stands for the folder itself, and @self reaches its contents. Aliases come from a luari.json (also accepted: luari, .luari) at the project root, and the extension watches it — alias edits apply live:

{
  "aliases": {
    "Settings": "./Scenes/Settings"
  }
}

export type Name = T makes a type visible to other files as modulevar.Name after local modulevar = require(...). Plain type aliases stay private. Anything declared pub — variables, classes, enums — is global to the whole project, exactly like the runtime's shared global scope.

Built-in type functions

These evaluate inside any type expression. They are tooling-only, like all type syntax, and the angle brackets follow the normal generic grammar.

FunctionResult
keyof<T>Union of T's field names as string literals. keyof<{ a: number, b: string }> is "a" | "b". Drives string autocompletion.
ValueOf<T, K>The type of field K in T. K may be a literal or a union of literals: ValueOf<T, keyof<T>> is the union of all field types.
ToBasic<T>Widens to the basic kind: literals to string/number/boolean, records to table, classes to class, enums to enum, functions to function. Unions widen member-wise.
NotNil<T>Strips nil from T (unions and T? optionals). A binding typed NotNil must be initialized non-nil and can never be assigned literal nil — enforced by the NotNil diagnostic above.

Both capitalisations are accepted (keyof/KeyOf, valueof/ValueOf, tobasic/ToBasic, notnil/NotNil).

Generics

local function identity<T>(x: T): T
  return x
end
local n = identity(5)        -- number

type veg = { Var: boolean, Ve: string }
local function get<i>(key: keyof<veg>): ValueOf<veg, i>
end
local v = get("Var")         -- boolean: i is bound from the argument

Generic aliases substitute their arguments (type Box<T> = { value: T }), and using a generic type with the wrong number of arguments is an error.

Self-referencing types and metamethods

A table type may reference itself, which makes chaining work:

type varg = {
  Input: (self: varg) -> varg
}
local t: varg = make()
t:Input():Input()            -- every link completes

Fields named after metamethods describe operator results, the same way class operator members do:

type Vec = {
  __add: (self: Vec, other: Vec) -> Vec,
  __sub: (self: Vec, other: number) -> number
}
local sum = a + b            -- Vec
local d = a - 5              -- number

Recognised: __add __sub __mul __div __mod __pow __concat __eq __lt __le __len __unm.

.luard ambient files

A file ending in .luard is written as normal LUAR code, but the runtime never loads it. Everything it declares — variables, functions, classes, enums, type aliases — becomes globally visible in every file of the project with no require. They exist so people can ship language-extension libraries: describe what a host engine injects, and the editor knows about it everywhere.

-- engine.luard
export type EntityId = number

class Entity {
  public id: number = 0
  function destroy()
  end
}

-- Spawns an entity by prefab name.
function spawn(prefab: string): Entity
end

TICK_RATE: number = 60

With that file anywhere in the project, every .luar file can call spawn("door") and get an Entity back, hover TICK_RATE, and use EntityId in annotations. Doc comments above the declarations appear on hover, and the declared names are semantically colored as builtins (see above). Function bodies can be empty — only the signatures matter. .luard files are not linted by Ferrite.

Mutability carries over. A const declaration (and function declarations, which are const by default) makes an immutable global: reassigning it in any .luar file is flagged MutateImmutable. A local declaration makes a mutable global, and the bare annotated form (TICK_RATE: number = 60) is also treated as mutable in a .luard file. A file that declares its own binding with the same name shadows the ambient global, and the local declaration's rules win.

Ambient files do not have to live in the project. The luard key in luari.json imports external .luard files by path — absolute (an engine's types kept in AppData) or relative to the config file. It accepts one path or a list:

{
    "aliases": { "Settings": "./Scenes/Settings" },
    "luard": [
        "C:/Users/you/AppData/Roaming/myengine/engine.luard",
        "./vendor/extra.luard"
    ]
}

When changes are picked up. Editing any .luard file inside VS Code re-analyzes the ambient globals live. luari.json is watched: changing the luard list (adding or removing a path) reloads the ambient set. But an external .luard file edited outside the editor is not watched — open it in VS Code, change the luard list, or run Luar: Restart Language Server to pick the edit up.

Declared auto-imports (Auto blocks)

A .luard file may contain Auto blocks — a feature exclusive to ambient files. Each entry pairs a name with an expression; the name then shows up in completion in every .luar file, and accepting it inserts a binding near the top of the file, the same way module auto-import inserts a require.

-- engine.luard
const game = {}

Auto {
    key "DS" Variable "game:GetService(\"DataStoreService\")"
    key "Net" Variable "game:GetService(\"NetworkService\")"
}

With that in place, typing DS in a script offers a completion item labeled auto-import — game:GetService("DataStoreService"). Accepting it inserts

local DS = game:GetService("DataStoreService")

after the file's require lines (or at the top when there are none), and leaves what you typed in place. The rules:

  • key is the variable name offered in completion; it must be a valid identifier. Variable is the exact expression text inserted on the right-hand side — escape quotes inside it with \".
  • Entries may be separated by newlines or commas, and several Auto blocks may appear across your .luard files. Duplicate keys keep the first entry.
  • A name that the current file already binds (a local, a function, a previous import) is not offered again, so accepting an entry twice cannot duplicate the line.
  • The block is editor-only syntax: it is stripped before the ambient file is type-checked, so it does not interfere with the other declarations around it. A malformed block is left in place and reported as a normal syntax error.

Ferrite lints in the editor

Every .luar file is checked live by Ferrite, the built-in code analyzer. Warnings (unused variables, shadowing, unreachable code, duplicate keys, and the rest of the check table) appear as squiggles alongside the editor's own diagnostics, with the check name appended to the message as [Code]. Ferrite's own MutateImmutable and SyntaxError are filtered out — the server's versions of those replace them. .luard ambient files are not linted.

Silence checks with the same directives the runtime API honors — the editor autocompletes both the directive keywords and the check names after --#:

--#disable UnusedVariable, ShadowedVariable  -- whole file
x = 2 --#disable-line MutateImmutable      -- just this line
--#disable-next-line ConstantCondition       -- the following line
--#disable all                               -- everything

Editor-only diagnostics are toggleable with the same syntax using these extra codes: DuplicateEnumVariant, GenericArity, UnknownClass, RequireCycle, NotNil, and FinalOverride.

Building the extension

cd Luar-LSP
cargo build --release
cp target/release/luar-lsp.exe editors/code/server/
cd editors/code && npm install && npm run build && npm run package
code --install-extension luar-lsp-<version>.vsix

The repository also has testbed/, a minimal Rust host that embeds the interpreter and runs scripts/main.luar with full require support — useful for trying language features end to end with cargo run.
