Host API (Rust)

LUAR is built to be embedded in a Rust program. You create an interpreter, inject globals, tables, native functions, and classes from Rust, run LUAR code against them, and read values back out. This section is a complete guide for building those things from the host side.

The Interpreter

The Interpreter is the entry point. It owns the global scope and keeps all state between calls, so you can run several pieces of code and have them see each other.

use luar::{Interpreter, Value};

let mut i = Interpreter::new();   // builtins and the standard library are preloaded
i.run_source("print(1 + 2)")?;       // run some code; state persists
i.run_source("print(1 + 2)")?;       // the next call sees everything the first set up

Convenience free functions for one-shot use:

let interp = luar::eval_source("print(\"hi\")")?;   // parse + run, returns the interpreter
let value  = Interpreter::load_module("return 42")?;  // run as a module, return its value

Starting a project

A real program is more than one string. The usual shape is: build an interpreter, point it at your project folder, inject whatever the host provides, then run an entry file that pulls in the rest with require.

use luar::{Interpreter, Value};

let mut i = Interpreter::new();
i.set_module_dir("./game");                  // relative require() paths resolve from here
i.set_global_const("VERSION", Value::str("1.0"));

let src = std::fs::read_to_string("./game/main.luar")?;
i.run_source(&src)?;                           // main.luar can require ./player, ./ui, ...

set_module_dir sets the base folder that relative require("./x") paths resolve against, so the entry file and everything it pulls in find each other no matter where the host binary was launched from. Each module is parsed and run once, then its result is cached.

Reloading modules

Because modules are cached, requiring the same path twice returns the same result. free_module drops a module from the cache and nils the values it published, so the next require runs the file fresh from disk — this is how you hot-reload a script while the program keeps running. It returns true if the module was loaded.

i.free_module("./player");                    // forget the cached ./player module
i.run_source("require(\"./player\")")?;         // re-runs ./player.luar from disk

Garbage collection

The interpreter collects garbage on its own as code runs. Call collect_garbage to force a collection at a natural pause, for instance right after unloading a large module.

i.collect_garbage();

Executing code

MethodWhat it does
i.run_source(src)Parse and run src in this interpreter. Returns any top-level return values.
i.call_value(&callable, args)Call a LUAR function or native value from Rust and get its results.
luar::eval_source(src)Run in a fresh interpreter and return it.
luar::run_precompiled(&bytes)Run a precompiled .luarc program.
// run code, then call a function it defined
i.run_source("function double(x) return x * 2 end")?;
let f = i.get_global("double").unwrap();
let out = i.call_value(&f, vec![Value::int(21)])?;   // [Value::Int(42)]

Launching coroutines from Rust

The host can launch any LUAR function — or an object method with self bound — as a coroutine. The launch resumes the coroutine immediately: if the body finishes (or yields) within the grace window you get the result right away; otherwise it keeps running in the background and the host drains it with luar::run_pending().

MethodWhat it does
i.launch(&callable, args)Create a coroutine from a function, class, or callable table and resume it with args. Returns the coroutine Value.
i.launch_method(&instance, "name", args)Launch instance:name(args) as a coroutine with self bound. Also works with a class value for static methods.
luar::run_pending()Block until every backgrounded coroutine has finished.
i.run_source(r#"
    class Player {
        public name: string = ""
        constructor(n) self.name = n end
        function respawn(delay)
            wait(delay)
            print(self.name .. " respawned")
        end
    }
    pub local hero = Player("ana")
    pub local tick = function(n) print("tick", n) end
"#)?;

let tick = i.get_global("tick").unwrap();
let co = i.launch(&tick, vec![Value::int(1)])?;          // plain function

let hero = i.get_global("hero").unwrap();
i.launch_method(&hero, "respawn", vec![Value::int(2)])?; // method, self bound

luar::run_pending();                                       // drain background work

Errors that happen immediately (a non-callable value, a missing method, or a body that fails inside the grace window) come back as the Err of the launch call. Launch only from the host thread — calling it while a coroutine is running is an error. The full pattern, including connecting and disconnecting script handlers signal-style, is runnable as cargo run --example signal in Luar-Runtime.

Yielding and resuming from Rust

A native function runs on the thread of whichever coroutine called it, so it can suspend just that coroutine without touching anything else:

MethodWhat it does
luar::yield_values(vals)Inside a native called from a coroutine: yield it with vals (they become the resume results). Returns the values passed to the next resume. Errors on the main thread.
luar::blocking(f)Inside a native called from a coroutine: run the closure while only THIS coroutine waits — every other coroutine and the main script keep running. Returns the closure's value to the native, which can hand it back to the script.
i.resume_coroutine(&co, args)Resume a Value::Coroutine from Rust (same results as coroutine.resume).
luar::current_coroutine()(thread, is_main) — the running coroutine, e.g. to validate a thread argument a script passed you.
// a wait-style native: parks ONLY the calling coroutine
fn my_wait(_i: &mut Interpreter, args: Vec<Value>) -> Result<Vec<Value>, String> {
    let secs = args.first().and_then(|v| v.as_float()).unwrap_or(0.0);
    luar::blocking(move || std::thread::sleep(std::time::Duration::from_secs_f64(secs)))?;
    Ok(vec![])
}

// a checkpoint native: yields the calling coroutine until the host resumes it
fn checkpoint(_i: &mut Interpreter, args: Vec<Value>) -> Result<Vec<Value>, String> {
    let sent_back = luar::yield_values(args)?;   // returns once the host resumes
    Ok(sent_back)
}

blocking is for "wait for something outside the script" (timers, IO, channels) — the host drains finished waiters with run_pending() / pump_ready(). yield_values is classic cooperative yielding: the coroutine stays suspended until someone resumes it (from the script or via resume_coroutine). The closure passed to blocking must not touch Luar values.

Tracking which script a value came from

Every require'd module is tagged with its file path automatically, and the host can tag the entry script too. Native functions can then ask who called them, and any function, table, or class remembers which script created it — useful for per-module registries, permissions, or debugging.

MethodWhat it does
i.set_source_path(path)Tag the entry script with its path (modules are tagged automatically).
i.current_source()Inside a native function: the path of the script currently running.
i.source_of_value(&v)The path of the script that created a function, table, or class.
i.script_of_value(&v)The creating script's id (pairs with free_script).
luar::script_source(id)Resolve a script id back to its path.
i.set_global_fn("register", |i, args| {
    let handler = args.into_iter().next().unwrap_or(Value::Nil);
    let who = i.source_of_value(&handler)
        .or_else(|| i.current_source());
    println!("handler registered by {:?}", who);   // e.g. mods/alerts.luar
    Ok(vec![])
});

Creating globals

Globals are visible to all code you run. Use set_global for a reassignable binding and set_global_const for an immutable one.

i.set_global("score", Value::int(0));             // local-like, reassignable
i.set_global_const("VERSION", Value::str("1.0"));  // const

i.run_source("score = score + 10  print(VERSION)")?;

Read a global back, and remove or clear one:

let score = i.get_global("score");   // Option<Value>
i.force_set("score", Value::int(0)); // set even if it is const, returns true if it existed
i.nil_value("score");                // set to nil and free what it held
i.remove_global("score");            // remove the binding entirely

Inspecting and editing variables mid-run

You look a global up by name with get_global, which returns Option<Value> — None means nothing by that name is defined right now, so the same call doubles as an existence check. There is no "list every variable" call; you query the names you care about and read the Value with the usual helpers.

match i.get_global("score") {
    Some(v) => println!("score = {}", v.as_int().unwrap_or(0)),
    None    => println!("score is not defined"),
}

Editing works the same whether the interpreter is idle between calls or in the middle of running a script. A native function receives &mut Interpreter, so it can read and rewrite globals while the code that called it is still on the stack; the change is visible to everything that runs afterward.

i.set_global_fn("doubleScore", |i, _args| {
    let cur = i.get_global("score").and_then(|v| v.as_int()).unwrap_or(0);
    i.force_set("score", Value::int(cur * 2));   // rewrite the global mid-execution
    Ok(vec![])
});

i.run_source(r#"
    score = 21
    doubleScore()      -- the host edits `score` while this runs
    print(score)       -- 42
"#)?;

The four editing calls differ only in how forceful they are: set_global declares or replaces a reassignable binding, force_set overwrites even a const (and returns whether it already existed), nil_value sets it to nil and frees what it held, and remove_global deletes the binding outright.

Creating native functions

A native function is a Rust function with the signature fn(&mut Interpreter, Vec<Value>) -> Result<Vec<Value>, String>. It receives the call arguments and returns result values (or an error message). Expose one as a global with set_global_fn.

i.set_global_fn("shout", |_i, args| {
    let s = args.first().map(|v| v.to_string()).unwrap_or_default();
    Ok(vec![Value::str(&s.to_uppercase())])
});

i.run_source("print(shout(\"hi\"))")?;   // HI

Creating tables and native libraries

Build a table with create_table, then fill it with set_field (any value) and set_native (a native function). This is how you expose a library table such as your own math or net.

let net = i.create_table();
net.set_field(Value::str("version"), Value::int(1))?;
net.set_native("get", |_i, args| {
    let url = args.first().map(|v| v.to_string()).unwrap_or_default();
    // ... do real work here ...
    Ok(vec![Value::str(&format!("fetched {url}"))])
});
i.set_global_const("net", net);

i.run_source("print(net.get(\"x\"))  print(net.version)")?;

You can also drive behavior with native metamethods. Attach __index, __newindex, __call, or __tostring to a metatable so host code runs when LUAR indexes, calls, or stringifies an object.

let meta = i.create_table();
meta.set_native("__index", |_i, args| {
    let key = args.get(1).map(|v| v.to_string()).unwrap_or_default();
    Ok(vec![Value::str(&format!("dynamic:{key}"))])
});
let obj = i.create_table();
obj.set_metatable(meta)?;
i.set_global_const("obj", obj);
i.run_source("print(obj.anything)")?;   // dynamic:anything

Creating classes

Define a class whose fields, methods, operators, getters, setters, and constructor are native Rust code with NativeClassBuilder, then register it as a global with define_class. Scripts can then call the class to construct instances and call its methods.

Every native callable on a class receives the instance as its first argument (this is self), followed by the call arguments.

use luar::{Interpreter, NativeClassBuilder, Value};

let mut i = Interpreter::new();
i.define_class(
    NativeClassBuilder::new("Counter")
        .field("count", Value::int(0))             // default field value
        .constructor(|_i, args| {                       // args[0] is self, then user args
            let start = args.get(1).cloned().unwrap_or(Value::int(0));
            args[0].set_field(Value::str("count"), start)?;
            Ok(vec![])
        })
        .method("bump", |_i, args| {                   // called as c:bump()
            let this = &args[0];
            let n = this.field(&Value::str("count")).as_int().unwrap_or(0) + 1;
            this.set_field(Value::str("count"), Value::int(n))?;
            Ok(vec![Value::int(n)])
        })
        .getter("value", |_i, args| {                  // read as c.value
            Ok(vec![args[0].field(&Value::str("count"))])
        })
        .metamethod("__tostring", |_i, args| {        // used by tostring/print
            let n = args[0].field(&Value::str("count"));
            Ok(vec![Value::str(&format!("Counter({n})"))])
        }),
);

i.run_source(r#"
    const c = Counter(10)
    c:bump()
    print(c:bump())   -- 12
    print(c.value)    -- 12
    print(c)          -- Counter(12)
"#)?;

Builder methods:

MethodAdds
.new(name)Start a class with the given name.
.extends(parent)Inherit from another class value: its fields carry over, and any member you do not redefine resolves through the parent. A class has exactly one parent.
.mixin(other)Fold in another class's members (methods, getters, setters, operators, fields) without inheritance. Your own definitions win.
.field(name, default)An instance field with a default value.
.method(name, fn)An instance method, called as obj:name(...).
.getter(name, fn) / .setter(name, fn)A property getter or setter, used as obj.name.
.metamethod(name, fn)An operator overload by metamethod name (__add, __eq, __tostring, ...).

.build()Produce the class value without registering it (use define_class to register).

Inheritance and mixins

.extends(parent) makes a native class inherit from another class value — its fields are carried over, and any method, getter, setter, or operator you do not redefine resolves through the parent chain. .mixin(other) folds in another class's members without inheritance. Your own definitions always take priority, and a class still has exactly one parent.

let animal = i.define_class(
    NativeClassBuilder::new("Animal")
        .field("name", Value::str("animal"))
        .method("describe", |_i, args| {
            Ok(vec![args[0].field(&Value::str("name"))])
        }),
);

i.define_class(
    NativeClassBuilder::new("Dog")
        .extends(animal)                            // inherits the name field and describe()
        .method("bark", |_i, _args| Ok(vec![Value::str("woof")])),
);

Scripts can extend a native class with the ordinary extends keyword too, so native and LUAR classes mix freely in one hierarchy.

Reacting to classes and instances

Often the host wants to know when script code defines or builds a class that descends from a base it provided — to drive a lifecycle, register the object, or wire it into the host. Two hooks deliver exactly that, matched by base class name anywhere in the ancestry.

HookWhen the callback fires
on_subclass_of(base, |i, class| ...)Whenever a script declares a class that descends from base. The callback receives the new class value.
on_instance_of(base, |i, instance| ...)After any instance whose class descends from base is constructed (its constructor has already run). The callback receives the new instance.
let mut i = Interpreter::new();
i.define_class(NativeClassBuilder::new("Entity").make_abstract());

i.on_subclass_of("Entity", |_i, _class| {
    // a script just defined a new kind of Entity — register or validate it
});
i.on_instance_of("Entity", |_i, instance| {
    // every Entity that gets built lands here — add it to your world,
    // stash a handle, kick off its lifecycle, and so on
    let _ = instance;
});

i.run_source(r#"
    class Player extends Entity { function start() print("ready") end }
    const p = Player()   -- fires on_subclass_of at the declaration, then on_instance_of
"#)?;

This is how you give a base class a lifecycle without baking one into the language: ship the base from Rust, then decide in host code what happens when scripts extend it or construct instances of it.

Working with Value

Construct values with Value::int, Value::str, Value::table, and the variants Value::Bool, Value::Float, Value::Nil. Read them with helpers such as as_int, to_string, type_name, field, and set_field. The native function signature is fn(&mut Interpreter, Vec<Value>) -> Result<Vec<Value>, String>, where the returned vector is the list of result values.
