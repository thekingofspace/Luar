Host API (Rust)

LUAR is built to be embedded in a Rust program. You create an interpreter, inject globals, tables, native functions, and classes from Rust, run LUAR code against them, and read values back out. This section is a complete guide for building those things from the host side.

Adding the dependency

The interpreter lives in the luar crate (the Luar/ folder of the repository). Point Cargo at it with a path dependency:

# Cargo.toml
[dependencies]
luar = { path = "path/to/Luar" }

A minimal embed parses and runs a script, then reads a global back out. Everything you need day to day is re-exported at the crate root.

use luar::{Interpreter, Value};

fn main() -> Result<(), luar::Error> {
    let mut i = Interpreter::new();
    i.run_source(r#"
        pub greeting = "hello from luar"
        print(greeting)
    "#)?;

    let g = i.get_global("greeting").unwrap();
    println!("rust sees: {g}");   // Value implements Display
    Ok(())
}

The Interpreter

The Interpreter is the entry point. It owns the global scope and keeps all state between calls, so you can run several pieces of code and have them see each other.

use luar::{Interpreter, Value};

let mut i = Interpreter::new();   // builtins and the standard library are preloaded
i.run_source("print(1 + 2)")?;       // run some code; state persists
i.run_source("print(1 + 2)")?;       // the next call sees everything the first set up

Convenience free functions for one-shot use:

let interp = luar::eval_source("print(\"hi\")")?;   // parse + run, returns the interpreter
let value  = Interpreter::load_module("return 42")?;  // run as a module, return its value
let ast    = luar::parse_source("return 42")?;     // just lex + parse (no run, no optimize)
let tokens = luar::tokenize("return 42")?;          // just lex

Errors: how failures surface

Every stage of running code reports through one enum, luar::Error. It implements Display and std::error::Error, and all five inner types convert into it with From, so ? works everywhere.

VariantProduced by
Error::Lex(LexError)Tokenizing — bad escapes, malformed numbers, unterminated strings.
Error::Parse(ParseError)Parsing — syntax errors.
Error::Eval(EvalError)The tree-walking interpreter — every runtime error, including Err(msg) returned by your native functions.
Error::Compile(CompileError)Bytecode compilation, and a bad .luarb header ("not a LUAR precompiled file").
Error::Runtime(RuntimeError)The experimental bytecode VM (run_on_vm / execute).

EvalError is a tuple struct wrapping the message string (EvalError(pub String)); it displays as eval error: <message>. Methods that go straight into the interpreter — call_value, call_method, launch, resume_coroutine — return Result<_, EvalError> directly; methods that also parse, such as run_source, return Result<_, luar::Error>.

match i.run_source("print(1 +") {
    Ok(_) => {}
    Err(luar::Error::Parse(e)) => eprintln!("syntax error: {e}"),
    Err(e) => eprintln!("script failed: {e}"),
}

An error raised by a native function travels the same path as error("...") raised in a script: it aborts the call unless a pcall catches it on the script side, and otherwise comes back to the host as Error::Eval.

i.set_global_fn("explode", |_i, _args| Err("boom".to_string()));

i.run_source(r#"
    local ok, msg = pcall(explode)
    print(ok, msg)        -- false   boom
"#)?;

let err = i.run_source("explode()").unwrap_err();   // Error::Eval("boom")

Starting a project

A real program is more than one string. The usual shape is: build an interpreter, point it at your project folder, inject whatever the host provides, then run an entry file that pulls in the rest with require.

use luar::{Interpreter, Value};

let mut i = Interpreter::new();
i.set_module_dir("./game");                  // relative require() paths resolve from here
i.set_source_path("./game/main.luar");       // tag the entry script (modules tag themselves)
i.set_global_const("VERSION", Value::str("1.0"));

let src = std::fs::read_to_string("./game/main.luar")?;
i.run_source(&src)?;                           // main.luar can require ./player, ./ui, ...

set_module_dir sets the base folder that relative require("./x") paths resolve against, so the entry file and everything it pulls in find each other no matter where the host binary was launched from. It is also where @alias names from a luari.json file are searched. Each module is parsed and run once, then its result is cached. set_source_path records the entry script's file path for the source-tracking APIs described below.

Reloading and unloading scripts

Because modules are cached, requiring the same path twice returns the same result. free_module drops a module from the cache and nils the values it published, so the next require runs the file fresh from disk — this is how you hot-reload a script while the program keeps running. It returns true if the module was loaded.

i.free_module("./player");                    // forget the cached ./player module
i.run_source("require(\"./player\")")?;         // re-runs ./player.luar from disk

Underneath, every function, table, and class remembers the numeric id of the script that created it. The id-level API lets a host unload any script, not just modules:

MethodWhat it does
i.current_script_id()This interpreter's own script id.
i.script_of_value(&v)The id of the script that created a function, table, or class (None for other kinds).
i.free_script(id)Kill every function the script created (nils them out of every scope and table) and nil the script's root variables. Returns true if the script was registered.
i.free_module(name)Resolve name like require would, evict it from the module cache, and free_script its scope.
i.has_live_functions(id)true while any function created by that script is still reachable.
i.live_function_count(id)How many of that script's functions are still alive.

Calling a function that belonged to a freed script raises attempt to call a function from a freed script, so stale handlers fail loudly instead of running dead code.

// run code, then call a function it defined

let f = i.get_global("double").unwrap();
let out = i.call_value(&f, vec![Value::int(21)])?;   // [Value::Int(42)]

Creating globals

Globals are visible to all code you run. Use set_global for a reassignable binding and set_global_const for an immutable one.

i.set_global("score", Value::int(0));             // local-like, reassignable
i.set_global_const("VERSION", Value::str("1.0"));  // const

i.run_source("score = score + 10  print(VERSION)")?;

Read a global back, and remove or clear one:

let score = i.get_global("score");   // Option<Value>
i.force_set("score", Value::int(0)); // set even if it is const, returns true if it existed
i.nil_value("score");                // set to nil and free what it held
i.remove_global("score");            // remove the binding entirely

Inspecting and editing variables mid-run

You look a global up by name with get_global, which returns Option<Value> — None means nothing by that name is defined right now, so the same call doubles as an existence check. There is no "list every variable" call; you query the names you care about and read the Value with the usual helpers.

match i.get_global("score") {
    Some(v) => println!("score = {}", v.as_int().unwrap_or(0)),
    None    => println!("score is not defined"),
}

Editing works the same whether the interpreter is idle between calls or in the middle of running a script. A native function receives &mut Interpreter, so it can read and rewrite globals while the code that called it is still on the stack; the change is visible to everything that runs afterward.

i.set_global_fn("doubleScore", |i, _args| {
    let cur = i.get_global("score").and_then(|v| v.as_int()).unwrap_or(0);
    i.force_set("score", Value::int(cur * 2));   // rewrite the global mid-execution
    Ok(vec![])
});

i.run_source(r#"
    score = 21
    doubleScore()      -- the host edits `score` while this runs
    print(score)       -- 42
"#)?;

The four editing calls differ only in how forceful they are: set_global declares or replaces a reassignable binding, force_set overwrites even a const (and returns whether it already existed), nil_value sets it to nil and frees what it held, and remove_global deletes the binding outright.

Creating native functions

A native function is a plain Rust function with this exact signature (luar::runtime::NativeFn):

fn(&mut Interpreter, Vec<Value>) -> Result<Vec<Value>, String>

It receives every call argument in order in the Vec<Value> (check args.first(), args.get(1), ... yourself — LUAR does not pad or check arity for you) and returns the result values; a function that returns nothing returns Ok(vec![]). Returning Err(message) raises a runtime error in the calling script, exactly like error(message) — catchable with pcall, otherwise fatal to the run. Expose a native as a global with set_global_fn.

i.set_global_fn("shout", |_i, args| {
    let s = args.first().map(|v| v.to_string()).unwrap_or_default();
    Ok(vec![Value::str(s.to_uppercase())])
});

i.run_source("print(shout(\"hi\"))")?;   // HI

Because NativeFn is a function pointer, the closure you pass must not capture variables (Rust only coerces capture-free closures to function pointers). To give a native persistent state, stash it in a thread_local!, or store it in a LUAR table or global the native reads back through the interpreter. The first parameter is the full &mut Interpreter, so a native can run more code, call script functions, and edit globals while it executes.

There is no as_float helper — to accept "a number" the way scripts mean it, match both numeric variants:

let n = match args.first() {
    Some(Value::Int(i))   => *i as f64,
    Some(Value::Float(f)) => *f,
    other => return Err(format!("expected a number, got {}",
        other.map(|v| v.type_name()).unwrap_or("nothing"))),
};

Creating tables and native libraries

Build a table with create_table (or Value::table() — both register the table with the garbage collector), then fill it with set_field (any value) and set_native (a native function). This is how you expose a library table such as your own math or net.

let net = i.create_table();
net.set_field(Value::str("version"), Value::int(1))?;
net.set_native("get", |_i, args| {
    let url = args.first().map(|v| v.to_string()).unwrap_or_default();
    // ... do real work here ...
    Ok(vec![Value::str(format!("fetched {url}"))])
});
i.set_global_const("net", net);

i.run_source("print(net.get(\"x\"))  print(net.version)")?;

The table helpers on Value are raw: set_field(key, v) writes without invoking __newindex, and field(&key) reads without __index (returning Nil when absent or when the value is not a table). Valid keys are integers, strings, booleans, and — by identity — tables, functions, classes, interfaces, and coroutines; whole floats collapse to integer keys, while nil and fractional floats are invalid. For bulk access you can also borrow the underlying Table directly:

if let Value::Table(rc) = &v {
    let t = rc.borrow();
    let n = t.len();                     // array-part length (what # reports)
    let first = t.get(&Value::int(1));   // raw read, Nil when absent
    for item in &t.array { println!("{item}"); }
}

You can also drive behavior with native metamethods. Attach __index, __newindex, __call, or __tostring to a metatable so host code runs when LUAR indexes, calls, or stringifies an object. set_metatable sets it (Value::Nil clears it).

let meta = i.create_table();
meta.set_native("__index", |_i, args| {
    let key = args.get(1).map(|v| v.to_string()).unwrap_or_default();
    Ok(vec![Value::str(format!("dynamic:{key}"))])
});
let obj = i.create_table();
obj.set_metatable(meta)?;
i.set_global_const("obj", obj);
i.run_source("print(obj.anything)")?;   // dynamic:anything

Creating classes

Define a class whose fields, methods, operators, getters, setters, and constructor are native Rust code with NativeClassBuilder, then register it as a global with define_class. Scripts can then call the class to construct instances and call its methods.

Every native callable on a class receives the instance as its first argument (this is self), followed by the call arguments.

use luar::{Interpreter, NativeClassBuilder, Value};

let mut i = Interpreter::new();
i.define_class(
    NativeClassBuilder::new("Counter")
        .field("count", Value::int(0))             // default field value
        .constructor(|_i, args| {                       // args[0] is self, then user args
            let start = args.get(1).cloned().unwrap_or(Value::int(0));
            args[0].set_field(Value::str("count"), start)?;
            Ok(vec![])
        })
        .method("bump", |_i, args| {                   // called as c:bump()
            let this = &args[0];
            let n = this.field(&Value::str("count")).as_int().unwrap_or(0) + 1;
            this.set_field(Value::str("count"), Value::int(n))?;
            Ok(vec![Value::int(n)])
        })
        .getter("value", |_i, args| {                  // read as c.value
            Ok(vec![args[0].field(&Value::str("count"))])
        })
        .metamethod("__tostring", |_i, args| {        // used by tostring/print
            let n = args[0].field(&Value::str("count"));
            Ok(vec![Value::str(format!("Counter({n})"))])
        }),
);

i.run_source(r#"
    const c = Counter(10)
    c:bump()
    print(c:bump())   -- 12
    print(c.value)    -- 12
    print(c)          -- Counter(12)
"#)?;

Builder methods (all of them — each consumes and returns the builder, so they chain):

MethodAdds
.new(name)Start a class with the given name.
.extends(parent)Inherit from another class value: its fields carry over, and any member you do not redefine resolves through the parent. A class has exactly one parent. Ignored unless parent is a Value::Class.
.mixin(other)Fold in another class's members (methods, getters, setters, operators, fields) without inheritance. Your own definitions win.
.field(name, default)An instance field with a default value. Only bool, int, float, and string defaults are kept — any other value becomes "no default" (set those in the constructor instead).
.method(name, fn)An instance method, called as obj:name(...).
.getter(name, fn) / .setter(name, fn)A property getter or setter, used as obj.name.
.metamethod(name, fn)An operator overload by metamethod name (__add, __eq, __tostring, ...).
.constructor(fn)A native constructor, run when the class is called.
.make_final() / .make_abstract()Mark the class final (cannot be extended) or abstract (cannot be instantiated).
.build()Produce the class value without registering it (use define_class to build and register the const global in one step).

Inheritance and mixins

.extends(parent) makes a native class inherit from another class value — its fields are carried over, and any method, getter, setter, or operator you do not redefine resolves through the parent chain. .mixin(other) folds in another class's members without inheritance. Your own definitions always take priority, and a class still has exactly one parent.

let animal = i.define_class(
    NativeClassBuilder::new("Animal")
        .field("name", Value::str("animal"))
        .method("describe", |_i, args| {
            Ok(vec![args[0].field(&Value::str("name"))])
        }),
);

i.define_class(
    NativeClassBuilder::new("Dog")
        .extends(animal)                            // inherits the name field and describe()
        .method("bark", |_i, _args| Ok(vec![Value::str("woof")])),
);

Scripts can extend a native class with the ordinary extends keyword too, so native and LUAR classes mix freely in one hierarchy.

Reacting to classes and instances

Often the host wants to know when script code defines or builds a class that descends from a base it provided — to drive a lifecycle, register the object, or wire it into the host. Two hooks deliver exactly that, matched by base class name anywhere in the ancestry. Unlike native functions, these callbacks are FnMut closures, so they may capture host state.

HookWhen the callback fires
on_subclass_of(base, |i, class| ...)Whenever a script declares a class that descends from base. The callback receives the new class value.
on_instance_of(base, |i, instance| ...)After any instance whose class descends from base is constructed (its constructor has already run). The callback receives the new instance.
let mut i = Interpreter::new();
i.define_class(NativeClassBuilder::new("Entity").make_abstract());

i.on_subclass_of("Entity", |_i, _class| {
    // a script just defined a new kind of Entity — register or validate it
});
i.on_instance_of("Entity", |_i, instance| {
    // every Entity that gets built lands here — add it to your world,
    // stash a handle, kick off its lifecycle, and so on
    let _ = instance;
});

i.run_source(r#"
    class Player extends Entity { function start() print("ready") end }
    const p = Player()   -- fires on_subclass_of at the declaration, then on_instance_of
"#)?;

This is how you give a base class a lifecycle without baking one into the language: ship the base from Rust, then decide in host code what happens when scripts extend it or construct instances of it.

Calling LUAR functions and methods from Rust

call_value calls anything callable — a script function, a native, a class (which constructs and returns an instance), or a table with a __call metamethod — and returns all of its results. call_method performs obj:name(args) dispatch on an instance, class (static methods), or table, respecting private/protected access and abstract checks just like script code. instance_has_method checks whether the receiver's class defines or inherits a method before you commit to calling it.

i.run_source(r#"
    function double(x) return x * 2 end
    class Greeter {
        function hello(name) return "hi " .. name end
    }
    pub g = Greeter()
"#)?;

let f = i.get_global("double").unwrap();
let out = i.call_value(&f, vec![Value::int(21)])?;            // [Int(42)]

let g = i.get_global("g").unwrap();
if i.instance_has_method(&g, "hello") {
    let r = i.call_method(&g, "hello", vec![Value::str("ana")])?;  // [Str("hi ana")]
}

Both return Result<Vec<Value>, EvalError>: every value the callee returned on success, or the runtime error it raised. Calls run to completion synchronously; for fire-and-forget calls that may wait, use launch below.

Async: coroutines, blocking, and the family GIL

One paragraph of model first. LUAR coroutines run on real OS threads, but a per-interpreter-family lock (the family GIL) guarantees only one of them executes LUAR code at any moment, so script state never races. When a native function needs to wait on the outside world it calls luar::blocking: the calling coroutine detaches, the GIL is released while the closure runs, and the coroutine re-attaches when the closure finishes — so a blocking native only blocks its own coroutine, while the main script and every other coroutine keep running. The host drives completion: luar::run_pending() blocks until every backgrounded coroutine has finished (the canonical last line of main), and luar::pump_ready() re-attaches whatever is ready right now and returns immediately. All the Interpreter convenience methods take the family GIL themselves, so they are safe to call between resumes.

Launching coroutines from Rust

The host can launch any LUAR function — or an object method with self bound — as a coroutine. The launch resumes the coroutine immediately: if the body finishes (or yields) within the grace window you get the result right away; otherwise it keeps running in the background and the host drains it with luar::run_pending().

MethodWhat it does
i.launch(&callable, args)Create a coroutine from a function, class, or callable table and resume it with args. Returns the coroutine Value.
i.launch_method(&instance, "name", args)Launch instance:name(args) as a coroutine with self bound. Also works with a class value for static methods.
i.resume_coroutine(&co, args)Resume a Value::Coroutine from Rust. Results follow the script convention: [true, ...yielded or returned values] on success, [false, message] on failure. Resuming a coroutine that is waiting inside blocking fails.
luar::run_pending()Block until every backgrounded coroutine has finished. No-op inside a coroutine.
luar::pump_ready()Re-attach coroutines that are ready right now, then return (non-blocking) — call it inside a host frame/tick loop.
i.run_source(r#"
    class Player {
        public name: string = ""
        constructor(n) self.name = n end
        function respawn(delay)
            wait(delay)
            print(self.name .. " respawned")
        end
    }
    pub local hero = Player("ana")
    pub local tick = function(n) print("tick", n) end
"#)?;

let tick = i.get_global("tick").unwrap();
let co = i.launch(&tick, vec![Value::int(1)])?;          // plain function

let hero = i.get_global("hero").unwrap();
i.launch_method(&hero, "respawn", vec![Value::int(2)])?; // method, self bound

luar::run_pending();                                       // drain background work

Errors that happen immediately (a non-callable value, a missing method, or a body that fails inside the grace window) come back as the Err of the launch call. Launch only from the host thread — calling it while a coroutine is running is an error. The full pattern, including connecting and disconnecting script handlers signal-style, is runnable as cargo run --example signal in Luar-Runtime.

Yielding and blocking from native code

A native function runs on the thread of whichever coroutine called it, so it can suspend just that coroutine without touching anything else:

FunctionWhat it does
luar::blocking(f)Inside a native called from a coroutine: run the closure while only THIS coroutine waits — every other coroutine and the main script keep running. Returns the closure's value to the native, which can hand it back to the script. On the main thread it just runs the closure (releasing the GIL around it and pumping ready coroutines).
luar::yield_values(vals)Inside a native called from a coroutine: yield it with vals (they become the resume results). Returns the values passed to the next resume. Errors on the main thread.
luar::current_coroutine()(thread, is_main) — the running coroutine as a Value, e.g. to validate a thread argument a script passed you, plus whether you are on the main thread.

This is the canonical wait native — the exact recipe from the testbed host. It parks ONLY the calling coroutine:

fn wait(_i: &mut luar::Interpreter, args: Vec<luar::Value>) -> Result<Vec<luar::Value>, String> {
    let secs = match args.first() {
        Some(luar::Value::Int(i))   => *i as f64,
        Some(luar::Value::Float(f)) => *f,
        _ => 0.0,
    };
    luar::blocking(move || std::thread::sleep(std::time::Duration::from_secs_f64(secs)))?;
    Ok(vec![])
}

// registration: i.set_global_fn("wait", wait);

// a checkpoint native: yields the calling coroutine until someone resumes it
fn checkpoint(_i: &mut luar::Interpreter, args: Vec<luar::Value>) -> Result<Vec<luar::Value>, String> {
    let sent_back = luar::yield_values(args)?;   // returns once the host resumes
    Ok(sent_back)
}

blocking is for "wait for something outside the script" (timers, IO, channels) — the host drains finished waiters with run_pending() / pump_ready(). yield_values is classic cooperative yielding: the coroutine stays suspended until someone resumes it (from the script or via resume_coroutine). The closure passed to blocking must not touch LUAR values — it runs while the GIL is released, and Value is not Send; compute plain data inside it and convert to values after it returns.

Tracking which script a value came from

Every require'd module is tagged with its file path automatically, and the host can tag the entry script too. Native functions can then ask who called them, and any function, table, or class remembers which script created it — useful for per-module registries, permissions, or debugging.

MethodWhat it does
i.set_source_path(path)Tag the entry script with its path (modules are tagged automatically).
i.current_source()Inside a native function: the path of the script currently running.
i.source_of_value(&v)The path of the script that created a function, table, or class.
i.script_of_value(&v)The creating script's id (pairs with free_script).
luar::script_source(id)Resolve a script id back to its path.
i.set_global_fn("register", |i, args| {
    let handler = args.into_iter().next().unwrap_or(Value::Nil);
    let who = i.source_of_value(&handler)
        .or_else(|| i.current_source());
    println!("handler registered by {:?}", who);   // e.g. mods/alerts.luar
    Ok(vec![])
});

The signal example in Luar-Runtime/examples/signal.rs uses exactly this: each connected handler stores source_of_value(&handler), so when the host fires the signal it can log which script file each handler came from — and free_script / free_module can later disconnect everything a given script registered.

Precompiling scripts

LUAR can serialize a parsed, optimized program to a compact binary .luarb file and run it later without reparsing — useful for shipping scripts without source, faster startup, and build pipelines. The format carries a versioned header; unpacking anything else fails with Error::Compile ("not a LUAR precompiled file (bad header)"). require loads .luarb files transparently (it tries name.luar first, then name.luarb).

FunctionWhat it does
luar::precompile_source(src)Parse + optimize + pack: returns the .luarb bytes (Vec<u8>).
i.run_precompiled(&bytes)Run precompiled bytes in this interpreter; returns the top-level return values.
luar::run_precompiled(&bytes)Run in a fresh interpreter and return it.
luar::run_precompiled_returns(&bytes)Run in a fresh interpreter and return (Interpreter, Vec<Value>) — the interpreter plus the script's return values.
luar::load_precompiled_module(&bytes)Run the bytes and return only the first returned value (or Nil) — module-style loading.
luar::precompile::pack(&stmts) / unpack(&bytes)The low-level AST ↔ bytes codec, if you manage parsing yourself.
// at build/deploy time: compile a module once
let src   = std::fs::read_to_string("mathx.luar")?;
let bytes = luar::precompile_source(&src)?;
std::fs::write("mathx.luarb", &bytes)?;

// at run time: load it as a module value and hand it to a host interpreter
let bytes  = std::fs::read("mathx.luarb")?;
let module = luar::load_precompiled_module(&bytes)?;   // the module's `return` value
i.set_global_const("mathx", module);

// or just execute a precompiled program in the current interpreter
i.run_precompiled(&bytes)?;

Loading modules from Rust

Interpreter::load_module is the source-string twin of load_precompiled_module: it runs the source in a fresh interpreter and gives you the module's first return value, which you can then inject into your host interpreter as a global.

let module = Interpreter::load_module(r#"
    local M = {}
    function M.greet(n) return "hi " .. n end
    return M
"#)?;
i.set_global_const("greetings", module);
i.run_source("print(greetings.greet(\"ana\"))")?;   // hi ana

For scripts loading each other, require does the work: relative paths resolve against set_module_dir (and against the requiring module's own folder), @alias names come from a luari.json file, resolution tries name.luar, name.luarb, then name/init.luar(b), every module runs once and is cached, and circular requires error with the full chain. If your modules do not live on disk at all — embedded resources, a database, a network — you can replace require wholesale, because it is just a global:

i.set_global_fn("require", |_i, args| {
    let name = args.first().and_then(|v| v.as_str()).unwrap_or_default().to_string();
    let source = my_module_source(&name)            // however the host stores scripts
        .ok_or_else(|| format!("no module named '{name}'"))?;
    let module = Interpreter::load_module(&source).map_err(|e| e.to_string())?;
    Ok(vec![module])
});

(The stock require caches per path; a custom one should keep its own cache — for instance in a thread_local! map — if modules must run only once. See Luar/examples/custom_require.rs for a complete version.)

Garbage collection

The interpreter collects garbage on its own: an allocation counter triggers a mark/sweep at the next safe point (between top-level statements). The host can force, request, and tune collection:

CallWhat it does
i.collect_garbage()Run a full mark/sweep right now, rooted at the interpreter's environment. Call it at a natural pause, e.g. right after unloading a large module.
luar::runtime::gc::request()Flag a collection to run at the next safe point (this is all the script-side collectgarbage() does).
luar::runtime::gc::set_threshold(n)Set how many allocations happen between automatic collections (default 10 000; clamped to at least 1). Lower = tighter memory, higher = fewer pauses.
luar::runtime::gc::live_objects()How many tables + functions are currently alive on this thread's heap — handy for leak hunting in tests.
use luar::runtime::gc;

gc::set_threshold(50_000);    // collect less often
i.run_source("...")?;
i.collect_garbage();          // force a sweep at a quiet moment
println!("live objects: {}", gc::live_objects());

The heap is thread-local: an Interpreter and every Value it creates must stay on the thread that created them (Value is not Send). Ownership is per-script — each object remembers the script that created it — which is what powers free_script and free_module above.

The optimizer

luar::optimize_program(&mut stmts) rewrites a parsed AST in place (for example, unrolling small numeric for loops with constant bounds). You almost never call it yourself: it runs automatically inside run_source, eval_source, precompile_source, compile_source, the run_precompiled family, and the require module loader. The two paths that stay raw are parse_source (tooling such as the LSP depends on the unmodified AST) and Context::spawn. So the only time to call it by hand is when you parse and execute manually:

let mut program = luar::parse_source(src)?;   // raw AST
luar::optimize_program(&mut program);          // optional, but what run_source would do
let results = i.run(&program)?;                // Interpreter::run never optimizes on its own

Context: hosting multiple cooperative scripts

luar::Context wraps one interpreter for hosts that juggle several scripts as resumable units — a game stepping many actor scripts, for example. spawn parses a source string into a coroutine that shares the context's global scope but does not start it; the host then steps it with resume whenever it likes.

MethodWhat it does
Context::new()A fresh context owning one interpreter.
ctx.run(src)Run source to completion (same as run_source).
ctx.spawn(src)Parse src into a not-yet-started coroutine sharing the context's globals; returns the coroutine Value.
ctx.resume(&co, args)Resume it: [true, ...yielded] or [false, message].
ctx.status(&co)"suspended", "running", "waiting", "dead" — or "not a thread".
ctx.set_global(name, v) / ctx.get_global(name)Shared globals, as on Interpreter.
ctx.interpreter()The underlying &mut Interpreter for everything else (natives, classes, modules).
use luar::{Context, Value};

let mut ctx = Context::new();
ctx.set_global("hp", Value::int(100));

let co = ctx.spawn(r#"
    while true do
        hp = hp - 10
        coroutine.yield(hp)
    end
"#)?;

let step = ctx.resume(&co, vec![])?;     // [true, 90]
let step = ctx.resume(&co, vec![])?;     // [true, 80]
println!("{}", ctx.status(&co));        // suspended
