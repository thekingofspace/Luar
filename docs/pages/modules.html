Modules

A module is simply a file. As your program grows you split it into several files, and one file uses another by requiring it. require loads and runs another file, then hands back whatever that file returned. Paths are relative to the file doing the requiring, and each module is run only once: the result is cached and reused on later requires.

local shapes = require("./shapes")   -- loads ./shapes.luar next to this file

The leading ./ means "next to this file". You can reach into subfolders ("./ui/button") or up a level ("../shared"). The .luar extension is added for you.

What is shared

pub declarations are published to the shared global scope, so any file that requires the module can use them. local and const declarations stay private to the module. A module can also return a value, which becomes the result of require.

-- greeting.luar
pub const SHARED_VERSION = "1.0"   -- visible everywhere once required
const SECRET = "hidden"            -- private to this module

local M = {}
function M.hello(name) return "hi, " .. name end
M.answer = 42
return M
-- main.luar
local g = require("./greeting")
print(g.hello("luar"))   -- the editor completes g.hello and g.answer
print(SHARED_VERSION)    -- available because it is pub

Folder modules: init.luar

A folder can be a module. If a folder contains a file named init.luar, you can require the folder by name and that init.luar is what runs. This lets you group a feature into its own folder and present a single entry point.

ui/
  init.luar      # this runs when you require("./ui")
  button.luar
  slider.luar
local ui = require("./ui")   -- runs ui/init.luar

When you require a path, LUAR looks for <path>.luar first, then falls back to <path>/init.luar.

require("@self"): a folder's children

Inside an init.luar (or any file), require("@self") returns a table of every sibling module in the same folder. Each name.luar file (except init.luar) and each subfolder that has its own init.luar becomes an entry, keyed by its name and already required. This makes it easy for an init.luar to gather up its folder's pieces.

-- ui/init.luar
local parts = require("@self")   -- { button = ..., slider = ... }
return {
  Button = parts.button,
  Slider = parts.slider,
}

Aliases: the .luarrc file

As a project grows, relative paths like "../../shared/util" become fragile and hard to read: move a file and every path that pointed near it breaks. A .luarrc file fixes this by letting you define short, stable names called aliases. An alias starts with an @ and stands in for a folder, so you write require("@shared/util") no matter where the requiring file lives.

There are no aliases built in. The .luarrc exists so that you (or whoever ships a project) can add the aliases that project wants. If you never make one, nothing changes.

Where to put it

Create a plain text file named exactly .luarrc (no extension), usually at the root of your project. When you require an @alias, LUAR starts from the folder of the file doing the requiring and walks up the folder tree until it finds a .luarrc that defines that alias. Because it searches upward, one .luarrc at the root serves the whole project, and a deeper .luarrc can add or override aliases for one part of it.

The format

One alias per line, written as name = path. The leading @ on the name is optional in the file. Blank lines are ignored, and a line beginning with # is a comment. The path is resolved relative to the folder that contains that .luarrc, so the same alias always points at the same place regardless of who requires it.

# .luarrc  (at the project root)
shared = ./src/shared
ui     = ./src/ui
this   = ./test
config = ./config

Using an alias

Now require those names from anywhere in the project. You can also tack a sub-path onto an alias.

local util = require("@shared/util")   -- ./src/shared/util.luar
local btn  = require("@ui/button")    -- ./src/ui/button.luar
local ui   = require("@ui")           -- ./src/ui/init.luar, if the folder has one

A full example

Given this layout:

myproject/
  .luarrc            # shared = ./src/shared
  main.luar
  src/
    game/
      player.luar    # require("@shared/math") works here too
    shared/
      math.luar
      init.luar      # makes @shared a folder module

Both main.luar and the deeply nested src/game/player.luar can write require("@shared/math"), and both resolve to src/shared/math.luar. The path never changes when you move the file that uses it. An alias follows the same rules as a normal require: it looks for <target>.luar first, then <target>/init.luar, so @shared on its own loads src/shared/init.luar.

Notes

  • If you require an alias that no .luarrc defines, you get an error telling you to define it.
  • The editor reads your .luarrc too, so completion and hover follow your aliases.
  • @self is not an alias you define; it is built in and always means the current folder's children (see above).

Module types

In the editor, a require binding tracks the imported module. Typing the module name and a dot offers the module's exported types (classes, enums, type aliases) and the fields of the table it returns. If no module of that name exists, the binding's type is shown as an error type.

Declaration files

A .luard file is a declaration-only file. Everything in it is treated as an ambient global for tooling, and it is never executed. Use it to describe globals the host provides from Rust, so the editor can complete and type them.

-- globals.luard
class Engine { public title: string = ""  function start() end }
enum LogLevel { Debug Info Warning Error }
local log: function = nil

Ferrite never lints a .luard file.
