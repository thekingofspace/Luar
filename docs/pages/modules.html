Modules

A module is simply a file. As your program grows you split it into several files, and one file uses another by requiring it. require loads and runs another file, then hands back whatever that file returned. Paths are relative to the file doing the requiring, and each module is run only once: the result is cached and reused on later requires.

local shapes = require("./shapes")   -- loads ./shapes.luar next to this file

The leading ./ means "next to this file". You can reach into subfolders ("./ui/button") or up a level ("../shared"). The .luar extension is added for you.

What is shared

pub declarations are published to the shared global scope, so any file that requires the module can use them. local and const declarations stay private to the module. A module can also return a value, which becomes the result of require — if it returns nothing, the requirer receives nil.

-- greeting.luar
pub const SHARED_VERSION = "1.0"   -- visible everywhere once required
const SECRET = "hidden"            -- private to this module

local M = {}
function M.hello(name) return "hi, " .. name end
M.answer = 42
return M
-- main.luar
local g = require("./greeting")
print(g.hello("luar"))   -- the editor completes g.hello and g.answer
print(SHARED_VERSION)    -- available because it is pub

How a path is resolved

For require("<path>"), LUAR tries these in order and loads the first that exists:

StepLooks for
1<path>.luar — a source file
2<path>.luarb — a precompiled file (see Precompilation)
3<path>/init.luar — a folder module
4<path>/init.luarb — a precompiled folder module

If none exists, the call fails with require: cannot find module '<path>'. A precompiled .luarb is a drop-in replacement for its source: anywhere require finds source, it accepts the packed form too.

Folder modules: init.luar

A folder can be a module. If a folder contains a file named init.luar (or init.luarb), you can require the folder by name and that init file is what runs. This lets you group a feature into its own folder and present a single entry point.

ui/
  init.luar      # this runs when you require("./ui")
  button.luar
  slider.luar
local ui = require("./ui")   -- runs ui/init.luar

Inside an init.luar, paths starting with a dot resolve from the folder's parent directory — the folder module behaves like a file sitting next to its folder, so require("./other") in ui/init.luar finds other.luar beside ui/, not inside it. To reach files inside its own folder, an init file requires them without the leading ./ (for example require("button")), or gathers them all with require("@self").

require("@self"): a folder's children

Inside an init.luar (or any file), require("@self") returns a table of every sibling module in the same folder. Each name.luar or name.luarb file (except the init file itself) and each subfolder that has its own init.luar(b) becomes an entry, keyed by its name and already required. Modules that are currently in the middle of being required are skipped, so an init.luar can safely gather up its folder's pieces.

-- ui/init.luar
local parts = require("@self")   -- { button = ..., slider = ... }
return {
  Button = parts.button,
  Slider = parts.slider,
}

Aliases: @name and luari.json

Relative paths get brittle in a deep tree — "../../engine/audio" breaks the moment a file moves. An alias names a folder once, in a config file, and every module can then require through it:

local audio = require("@engine/audio")   -- alias plus a sub-path
local eng   = require("@engine")         -- or the alias root itself

Aliases are defined in a luari.json file (the names luari and .luari work too). Paths are relative to the folder containing the config file:

{
  "aliases": {
    "engine": "./src/engine",
    "vendor": "./third_party"
  },
  "luard": ["./types/host.luard"]
}

When a module requires "@engine/...", LUAR walks up the directory tree from that module's folder, checking each level for luari.json / luari / .luari, until it finds one that defines the alias. A project usually keeps a single config at its root; nested configs simply win for the files beneath them. An alias nobody defines fails with require: unknown alias '@engine' (define it in a luari.json file).

The "luard" key in the example is read by the editor, not the runtime: it lists extra .luard ambient declaration files (a path or an array of paths, relative to the config) whose declarations the language server treats as globals in every file. The runtime only uses "aliases".

If no luari.json variant defines the alias at some level, a plain-text .luarrc file is also consulted there: one @name = path (or name = path) per line, with # for comments.

# .luarrc — same idea, plain text
@engine = src/engine
@vendor = third_party

The module cache

Each module runs once. The first require executes the file and caches its returned value under the file's canonical path; every later require of the same file — through any spelling, relative path, or alias that reaches it — returns the same cached value without running anything. This is what makes a module table a safe place for shared state, and why top-level side effects (prints, registrations) happen only on the first require.

The cache lives for the whole program. A Rust host can evict one module with free_module(name), which also tears down the module's functions — see Garbage collection; the next require then runs the file fresh.

Require cycles

If module A requires B while B is (directly or indirectly) requiring A, the require fails immediately rather than looping forever. The error spells out the whole chain:

require: circular require of './a' (a.luar -> b.luar -> a.luar)

The usual fix is to move the shared piece into a third module both can require, or to pass the value in as an argument instead of requiring it back. The editor also flags cycles as you type, before anything runs.
